The "Lost Update" Problem

1.1 The Conflict

What happens when two threads access the same piece of data at the same time?

Race Condition

Imagine two people open the same Excel file on a shared drive, edit it locally, and hit "Save". The last person to save wipes out the other's work.

1.2 Visualizing the Race

Even a simple counter++ operation isn't atomic. It breaks down into three distinct CPU instructions:

  1. Read count from memory to register.
  2. Add 1 to register.
  3. Write register back to memory.

If Thread A and Thread B both read the value "5" at the exact same time, they will both increment their local registers to "6" and both write "6" back to memory. Two increments happened, but the counter only went up by 1. One update was completely lost.

Characteristics of Race Conditions:

  • Non-deterministic: Different runs produce different results.
  • Timing-dependent: Depends on exactly how the OS scheduler interleaves the threads.
  • Extremely hard to debug: Bugs may be intermittent and hardware-dependent (only showing up on multicore systems).
2

Critical Sections & Mutex

2.1 Mutual Exclusion

A Critical Section is the specific lines of code that access shared resources.

The solution to race conditions is Mutual Exclusion (Mutex).

The Airplane Bathroom Analogy

Only one person fits inside. If the door shows "Occupied" (Locked), you wait in the aisle (Block). When they leave, the sign turns "Vacant" (Unlock), and the next person can enter.

2.2 The Pthreads Mutex API

// Step 1: Declare
pthread_mutex_t lock;

// Step 2: Initialize
pthread_mutex_init(&lock, NULL);

// Step 3: Protect critical section
pthread_mutex_lock(&lock);   // LOCK
counter++;                   // CRITICAL SECTION
pthread_mutex_unlock(&lock); // UNLOCK

// Step 4: Destroy when done
pthread_mutex_destroy(&lock);

Performance Note: Locking is safe, but it makes your code slower because it forces threads to wait.

3

Process Context Switching

3.1 The Trigger

A Context Switch is the mechanism that allows the CPU to switch from running Process A to Process B. It is triggered by:

  • Timer Interrupt (Preemptive multitasking).
  • I/O Block (Voluntary yielding).

3.2 Anatomy of a Switch (The Heavy Lift)

  1. Trap/Interrupt: Switch from User Mode to Kernel Mode.
  2. Save State: Registers, Program Counter, and Stack Pointer are saved to Process A's PCB.
  3. Scheduler: Decides "Who runs next?".
  4. Memory Switch: Switch Page Tables (CR3 register). This is extremely expensive!
  5. Restore State: Load Process B's registers.

3.3 The Hidden Costs

While saving registers takes ~200 cycles (visible cost), the real cost is invisible: Cache Pollution & TLB Flush (~5000+ cycles).

The TLB (Translation Lookaside Buffer) caches virtual to physical memory mappings. Since Process A's addresses are meaningless to Process B, the OS must completely flush (empty) the TLB.

Consequence

The new process starts with a "cold" brain. Every memory access is slow until the TLB and L1/L2 caches refill with the new process's data.

4

Threads: The Lightweight Alternative

4.1 Process vs Thread Switch

Because threads belong to the same process, switching between two threads in the same process is much faster (estimated ~0.5 microseconds vs ~3-5 microseconds for a full process switch).

  • What they SHARE (Don't need to switch): Page Tables, TLB entries (virtual addresses are still valid!), L3/L2 Cache data.
  • What they SWITCH (Must save/restore): CPU Registers, Stack Pointer, Program Counter.
5

The Scheduler vs Dispatcher

5.1 The Scheduler (The Brain / The Manager)

This is pure software logic. It does not touch hardware registers. It simply looks at the "Ready Queue" and runs an algorithm to pick the PCB of the next process.

Analogy: The Restaurant Manager deciding which table gets their food next.

5.2 The Dispatcher (The Muscle / The Worker)

This is low-level assembly code. It executes the actual hardware manipulation (saving registers, flushing TLB, jumping to the new program counter).

Analogy: The Waiter physically clearing the old plates, wiping the table, and setting down the new food. This physical work takes time and energy.

6

The Trade-off: When to Switch?

6.1 Throughput vs Latency

The scheduler must decide the Time Quantum (how long a process runs before it is preempted).

  • Small Quantum (e.g. 1ms): Switching too frequently causes high overhead. The computer feels highly responsive, but little actual work gets done (Low Throughput) because the CPU spends too much time "clearing the table".
  • Large Quantum (e.g. 500ms): Switching too infrequently causes high latency. The CPU is very efficient (high throughput), but the user experience is terrible (mouse stutters).

The Sweet Spot: Modern OSs use dynamic quantums (usually 10-20ms). It's fast enough for human eyes (<16ms) but keeps overhead low (<1%).

6.2 Trigger Code Example

void timer_interrupt_handler() {
    // 1. Save state
    save_context(&current->context);
    
    // 2. Accounting
    current->quantum_remaining--;
    
    // 3. Preemption logic
    if (current->quantum_remaining == 0) {
        enqueue_ready(current);
        
        // 4. Scheduler (Brain)
        process_t *next = scheduler_select_next();
        
        // 5. Dispatcher (Muscle)
        dispatch(next);
    }
}
