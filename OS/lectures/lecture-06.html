The Multitasking Trick

Your computer likely has 4 to 8 CPU cores, yet Activity Monitor shows 200+ processes running simultaneously. Music plays, browsers load, and code compiles—all at once. How is this possible?

1.1 The Restaurant Analogy

Too Many Guests, Not Enough Tables

Imagine a restaurant with 4 tables (representing your CPU cores) but 100 customers waiting in line (processes). The Maitre d' must strategically decide who sits next, for how long, and how to keep everyone happy. This is the essence of CPU Scheduling.

1.2 What is CPU Scheduling?

The OS acts as a Traffic Controller executing 10 to 100 times per second to:

  • Decide which process runs next.
  • Allocate CPU time fairly.
  • Maximize system performance and keep interactive apps responsive.
2

When Scheduling Happens

2.1 The Four Critical Moments

The scheduler must make a decision when one of four things happens in the process state lifecycle:

  1. Process blocks: Running → Waiting (It needs I/O, so let someone else use the CPU).
  2. Process finishes: Running → Terminated.
  3. Time expires: Running → Ready (The time quantum ran out, Preemption occurs).
  4. I/O completes: Waiting → Ready (A process returns to the ready queue).
3

What Makes Good Scheduling?

3.1 Performance Metrics

You cannot optimize for everything simultaneously. Here are the competing goals:

  • CPU Utilization: Keep the CPU busy. (Busy Time / Total Time * 100%).
  • Throughput: Maximize the number of processes completed per second.
  • Turnaround Time: Minimize the time from Arrival to Completion.
  • Waiting Time: Minimize the time a process spends sitting in the Ready Queue.
  • Response Time: Minimize the time from Arrival to the First CPU execution. Critical for UI snappiness!

3.2 Batch vs Interactive

Different Systems, Different Priorities
  • Servers (Batch): Prioritize Throughput and CPU Utilization.
  • Laptops/Phones (Interactive): Prioritize Response Time and Fairness.
  • Aircraft/Medical (Real-Time): Prioritize meeting hard deadlines with extreme predictability.
4

Preemptive vs Non-Preemptive

4.1 The Time Quantum

Non-Preemptive: A process runs until it finishes or voluntarily blocks. One long process can monopolize the CPU. It is simple but terrible for interactive systems.

Preemptive: The OS can forcibly take the CPU away from a running process using a Timer Interrupt. This enables fair sharing and good response times. All modern systems use this.

Preemption's Secret Weapon

The Time Quantum (or Time Slice) is a fixed amount of CPU time given to each process (typically 10-100ms). If it's too large, the system behaves non-preemptively. If it's too small, context-switching overhead wastes all the CPU power.

5

Scheduling Queues & Processes

5.1 The Three Key Queues

  • Ready Queue: Waiting for the CPU. (The most important queue).
  • I/O Queues: Waiting for devices (disk, network).
  • Job Queue: All processes currently in the system.

5.2 CPU-Bound vs I/O-Bound

Processes constantly alternate: [CPU burst] -> [I/O burst] -> [CPU burst] ...

  • CPU-Bound: Long CPU bursts (100ms+). Things like video encoding or simulations. They need long quantums and high throughput.
  • I/O-Bound: Short CPU bursts (1-10ms). Things like text editors or terminals. They need quick response times and high priority.

Most processes are I/O bound. The perfect mix keeps both the processor and the devices busy simultaneously.

6

Five Ways to Schedule

There is no universal "best" algorithm. Each optimizes different goals.

6.1 FCFS (First-Come, First-Served)

Processes are executed strictly in their arrival order. It is very simple and guarantees no starvation.

The Convoy Effect

If a massive 24-second process arrives first, all the tiny 3-second processes behind it must wait 24 seconds. This leads to terrible average waiting times.

6.2 SJF (Shortest Job First)

Executes the process with the shortest burst time next. This provides the mathematical optimal average waiting time.

The Catch: It is impossible to predict the exact burst length of a process beforehand. It can also lead to starvation for long processes if short processes keep arriving.

6.3 Round Robin (RR)

Perfect for interactive systems! Every process gets a small, fixed time slice (quantum). If it doesn't finish, it gets pushed to the back of the queue.

Pros: Extremely fair, good response time, no starvation. Cons: Higher turnaround time and high context-switching overhead if the quantum is too small.

6.4 Priority & Multilevel

  • Priority: The highest priority process runs first. Real-time tasks get Priority 1. Risk: Starvation for low-priority tasks. Fix: Aging (gradually increasing the priority of waiting tasks).
  • Multilevel Queue: Used by modern systems like macOS. It creates multiple queues with different rules (e.g., Queue 1 = Real-time FCFS, Queue 2 = Interactive RR, Queue 3 = Batch RR). macOS uses 128 priority levels!
7

Summary & Takeaways

7.1 Key Takeaways

Summary
  • Scheduling is the OS traffic controller.
  • Different goals exist for different system types (Throughput vs Response Time).
  • Preemption enables fair sharing, which is the modern standard.
  • Process types (CPU/I/O bound) drastically affect scheduling decisions.
  • There is no perfect algorithm - all involve trade-offs. Modern OSs use sophisticated multilevel scheduling to balance these out.
