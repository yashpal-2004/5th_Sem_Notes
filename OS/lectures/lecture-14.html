2.2 Producer-Consumer Problem - Introduction

What is the Producer–Consumer Problem?

○ Producer → creates data, puts into buffer ○ Consumer → removes data, processes it

Real-World Examples Print Spooler: Apps produce jobs → Printer consumes Web Server: Server produces pages → Browser consumes Assembly Line: Production → Packaging Message Queue: Sender → Receiver

Key Idea conditions

  • Also called the Bounded Buffer Problem
  • A classic synchronization problem
  • Involves two types of processes:
  • Shared buffer requires synchronization
  • Prevents: Buffer overflow. Buffer underflow. Race

2.3 Problem Statement

Bounded Buffer Scenario Buffer: Fixed-size array with N slots Producers: Generate data items Consumers: Consume data items

Operations Producer inserts item at in Consumer removes item from out Buffer works as a circular (ring) buffer

Conceptual View Producer → [ I1 ][ I2 ][ I3 ][ _ ][ _ ] → Consumer

Key Points Overflow (buffer full).

Underflow (buffer empty)

  • Requires synchronization

2.4 Synchronization Requirements

1⃣ Mutual Exclusion Only one process accesses buffer at a time 2⃣ Producer Must Wait Buffer is FULL 3⃣ Consumer Must Wait Buffer is EMPTY

Additional Consideration Avoid holding resources unnecessarily

3.2 Semaphore-Based Solution - Overview

Semaphores Required 1⃣ mutex (Binary Semaphore) 2⃣ empty (Counting Semaphore) 3⃣ full (Counting Semaphore)

Key Insight

  • Counting semaphores → track resource availability (empty, full)
  • Binary semaphore → ensures safe access (mutex)
  • Together → correct & safe synchronization

3.3 Producer Process - Code

Producer Algorithm int item;

item = produce_item(); // Produce item (outside CS)
buffer[in] = item; // Insert item
in = (in + 1) % N; // Circular buffer update

Key Points Circular buffer using (in + 1) % N

void producer() {
while (TRUE) {
wait(empty);             // Wait for empty slot
wait(mutex);             // Enter critical section
signal(mutex);           // Exit critical section
signal(full);            // One more full slot
}
}
wait(empty) → ensures buffer is not full
wait(mutex) → mutual exclusion
signal(full) → notifies consumer

4.1 Consumer Process - Code

Consumer Algorithm int item;

item = buffer[out]; // Remove item
out = (out + 1) % N; // Circular buffer update

consume_item(item); // Consume item (outside CS)

Key Points Circular buffer using (out + 1) % N

void consumer() {
while (TRUE) {
wait(full);             // Wait for available item
wait(mutex);            // Enter critical section
signal(mutex);          // Exit critical section
signal(empty);          // One more empty slot
}
}
wait(full) → ensures buffer is not empty
wait(mutex) → exclusive buffer access
signal(empty) → notifies producer

Common Code Pitfalls & Wrong Ordering

5.1 Common Mistake - Wrong Semaphore Order

Incorrect Producer Code item = produce_item(); buffer[in] = item;

in = (in + 1) % N;

Why This Causes Deadlock Deadlock: both wait forever

Root Cause

Correct Rule: Wait for resources first. Then acquire mutex. Release mutex before signaling

  • Producer locks mutex
  • Consumer cannot acquire mutex
  • Holding mutex while waiting for resources
  • Violates proper semaphore ordering
void producer() {
wait(mutex);
wait(empty);
signal(mutex);
signal(full);
}
● Buffer is full → blocks on wait(empty)
● Consumer cannot execute signal(empty)

5.2 Variations - Multiple Producers/Consumers

Multiple Producers item = produce_item(); buffer[in] = item;

in = (in + 1) % N;

Multiple Consumers item = buffer[out];

out = (out + 1) % N;

consume_item(item);

Same Solution Applies: Works for 1 or many producers. Works for 1 or many consumers. No code changes required.

Why It Works Semaphores manage waiting queues Multiple processes can block on same semaphore FIFO ordering ensures fairness No race conditions or deadlock

void producer_i() {
while ( TRUE) {
wait(empty);
wait(mutex);
signal(mutex);
signal(full);
}
}
void consumer_j() {
while ( TRUE) {
wait(full);
wait(mutex);
signal(mutex);
signal(empty);
}
}

5.3 Variation - Unbounded Buffer

Semaphores Used

semaphore mutex = 1; // Mutual exclusion
semaphore full = 0; // Items available

// No 'empty' semaphore

Producer item = produce_item(); append_to_buffer(item);

Consumer item = remove_from_buffer(); consume_item(item);

Key Differences (vs Bounded Buffer) No empty semaphore Producer never blocks (infinite space) Consumer blocks only when buffer is empty mutex still required for safety

When Buffer Has No Size Limit: Buffer has infinite capacity. No overflow possible. Synchronization becomes simpler.

void producer() {
while ( TRUE) {
wait(mutex);
signal(mutex);
signal(full);
}
}
void consumer() {
while ( TRUE) {
wait(full);
wait(mutex);
signal(mutex);
}
}

6.1 Performance Considerations

Key Performance Metrics Throughput → items processed per second Latency → time from produce to consume CPU Utilization → useful work vs waiting 20

Optimizing the Solution 1⃣ Minimize Critical Section Keep only buffer access inside lock Move produce_item() / consume_item() outside 2⃣ Avoid Busy Waiting Use blocking semaphores (sleep / wakeup) Avoid spinlocks for long waits 3⃣ Buffer Size Selection Large buffer → higher throughput, more memory Small buffer → less memory, more blocking Choose size based on producer–consumer rate 4⃣ Lock Granularity Coarse-grained lock → simple, less concurrency Fine-grained locks → more concurrency, complex design

Core Semaphores

Golden Rules

  • mutex = 1 → Mutual exclusion
  • empty = N → Empty buffer slots
  • full = 0 → Filled buffer slots
  • Wait for resources before mutex
  • Keep critical sections small
  • Always test with multiple producers & consumers
