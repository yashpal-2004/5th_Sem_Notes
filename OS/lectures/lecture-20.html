1.3 Problems with Contiguous Allocation

02

🧩

External Fragmentation

📐

Limited Flexibility

Hard to grow memory dynamically

📉

Poor Memory Utilization

🔄

Compaction Overhead

Costly: O(n) time complexity

📚

Difficult Memory Sharing

Wastes memory

🚀

Key Takeaway

Contiguous allocation is inefficient and inflexible. Need a non-contiguous memory allocation technique (Paging)

  • Free memory split into small holes
  • Enough total space, but not contiguous
  • Example: 50KB + 30KB + 40KB ≠ 100KB
  • Process size must be known in advance
  • ❌
  • Free space wasted due to fragmentation
  • Leads to inefficient RAM usage
  • Processes must be moved
  • System pause required
  • ⏱
  • Shared libraries get duplicated
  • ❌

2.1 Introduction to Paging

03

What is Paging?

🧠

Core Idea

Any page can be placed in any free frame

📦

Page / Frame Size

⭐

Key Advantage

No External Fragmentation

Better memory utilization

Flexible process allocation

  • Paging is a memory management technique
  • Allows a process to be stored in non-contiguous physical memory
  • Removes the need for contiguous allocation
  • Physical Memory → Frames (fixed-size blocks)
  • Logical Memory → Pages (same size as frames)
  • ✅
  • Fixed size, power of 2
  • Common sizes: 4 KB (most common). 8 KB. 4 MB
  • Power of 2 → easy address calculation
  • ❌
  • ✔
  • ✔

2.3 Frame Table & Page Table

05

🧱

1. Frame Table (Maintained by OS)

One entry per physical frame

Tracks allocated / free frames

Stores:

Frame number

Status (Free / Used)

Process ID (owner)

📌

Purpose: Efficient physical memory management

📘

2. Page Table (One per Process)

One entry per logical page

Maps Page → Frame

Stored in main memory

Base address stored in PCB

📌

Purpose: Address translation

06

🧩

PTE – What It Contains

┌────────┬───────┬─────────┬──────────┬───────────┬───────────┐

│ Frame │ Valid │ Protect │ Dirty │ Referenced│ Cache │

│ Number │ Bit │ (R/W/X) │ Bit │ Bit │ Disable │

└────────┴───────┴─────────┴──────────┴───────────┴───────────┘

🔍

Field Overview

○ Size depends on RAM (e.g., 18 bits for 1GB

RAM, 4KB frames)

○ 1 → Page present in memory

○ 0 → Page fault occurs

○ Example: 100 = Read-only, 110 = Read–Write

○ Used during page replacement

○ Helps replacement algorithms (LRU, Clock)

○ Used for memory-mapped I/O

  • Frame Number
  • Valid / Invalid Bit
  • Protection Bits (R / W / X)
  • Dirty (Modified) Bit
  • Referenced Bit
  • Cache Disable Bit

3.2 Address Translation - The Big Picture

07

🧠

Logical Address

┌──────────────────┬──────────────┐

│ Page Number (p) │ Offset (d) │

└──────────────────┴──────────────┘

(m − n bits) (n bits)

🧮

Example

Logical Address ( 16 bits)

→ Page Number: 6 bits

→ Offset: 10 bits💾

Physical Address

┌──────────────────┬──────────────┐

│ Frame Number (f) │ Offset (d) │

└──────────────────┴──────────────┘

📌

Key Rule:

👉

Only page → frame is translated

👉

Offset is unchanged

  • Process generates a logical address (m)
  • Split into Page Number + Offset
  • Logical address space = 2¹⁶ = 64 KB
  • Page size = 2¹⁰ = 1 KB
  • Number of pages = 64
  • Page number bits = 6
  • Offset bits = 10
  • Formed after page table lookup
  • Frame Number replaces Page Number
  • Offset stays the same

3.3 Address Translation - Step by Step

08

🧠

Big Flow

CPU

Logical Address (p, d)

↓

Page Table Lookup (p → f)

↓

Physical Address (f, d)

🧩

Step-by-Step Translation

1. Extract Page Number (p)

○ p = Logical Address bits - d

2. Extract Offset (d)

○ d = log₂(Page Size)

3. Page Table Lookup

○ Use p to index page table

○ Check Valid bit

■ ❌

Invalid → Page Fault (OS intervention)

■ ✔

Valid → Get frame number f

4. Form Physical Address

○ Physical Address = (f × Page Size) + d

5. Access Memory

○ CPU reads/writes at computed physical address

5.1 Page Table Storage Problem

12

⚠

The Challenge (32-bit System)

📌

100 processes → 400 MB only for page tables!

🚨

Even Worse in 64-bit Systems

Completely impractical

🧠

Why This is a Problem

✅

Practical Solutions

  • Logical address space: 4 GB (2³²)
  • Page size: 4 KB (2¹²)
  • Pages for process: 2²⁰ ≈ 1 million
  • PTE size: 4 bytes
  • Page table size = 4 MB per process
  • Pages ≈ 2⁵²
  • Page table size → Petabytes per process
  • ❌
  • Wastes huge memory
  • Most page table entries unused
  • Poor scalability
  • Multi-Level (Hierarchical) Page Tables
  • Hashed Page Tables
  • Inverted Page Tables
