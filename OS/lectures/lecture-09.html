CPU Scheduling Evaluation

1.1 The Scheduler's Dilemma

You're the OS scheduler. Five processes arrive:

Question: Which runs first? Can one algorithm rule them all?

  • Emergency System Update (2 seconds, CRITICAL)
  • Student's 10-hour Rendering (just started)
  • Video Call (needs CPU every 50ms)
  • File Download (quick bursts between waits)
  • Background Backup (can wait, but needs to finish)

1.2 Quick Check - Algorithm Selection

ANSWER: C) Round Robin - ensures fairness and good response time for all students. In a shared system, preventing any student from monopolizing CPU is crucial.

If you had to choose ONE algorithm for a university's time-sharing system with 100 students, which would you pick and why?

A) FCFS B) SJF C) Round Robin D) Priority

1.3 THE EVALUATION FRAMEWORK

8 Critical Metrics:

1. CPU Utilization - Keep the engine running (40-90%) 2. Throughput - How many jobs completed per time unit?

3. Turnaround Time - Start to finish duration 4. Waiting Time - Frustration in the ready queue 5. Response Time - Time to first acknowledgment (UX critical!) 6. Fairness - Everyone gets a turn 7. Predictability - Consistent performance 8. Overhead - Cost of context switches and management

1.4 ALGORITHM QUICK RECAP

  • FCFS (First-Come, First-Served)
    • Simple, non-preemptive
    • Convoy effect problem
    • Low overhead, predictable
  • SJF/SRTF (Shortest Job First)
    • Optimal average waiting time
    • Requires burst time prediction
    • Starvation risk for long jobs
  • Round Robin
    • Time quantum-based fairness
    • Excellent response time
    • Context switch overhead
  • Priority Scheduling
    • Importance-based allocation
    • Starvation risk (solved by aging)
    • Good for real-time requirements

1.5 ALGORITHM CHAMPIONSHIP

Key Insight: There's no clear winner! Each algorithm is like a specialized tool.

1.6 WORKLOAD CHARACTERISTICS

CPU-Bound Processes: encoding, 3D rendering efficiency I/O-Bound Processes: database queries responsiveness Mixed Workload:

  • Long CPU bursts, rare I/O
  • Examples: Scientific computing, video
  • Best algorithm: SJF/SRTF for
  • Short CPU bursts, frequent I/O waits
  • Examples: Text editors, web browsers,
  • Best algorithm: RR or Priority for
  • Combination of both types
  • Most realistic scenario in modern systems
  • Requires balanced approach (MLFQ)

Multi-Level Feedback Queues (MLFQ)

3.1 Test Your Understanding

Which scheduling algorithm provides optimal average waiting time?

A) FCFS B) SJF C) Round Robin D) Priority ANSWER: B) SJF This is mathematically proven - always run the shortest job next minimizes average waiting time.

3.2 What If We Use Them ALL?

The Idea: Use different algorithms for different process types!

3.3 FIXED MULTI-LEVEL QUEUES

How It Works:

1. Classify processes on arrival (system/interactive/batch) 2. Each queue has its own algorithm 3. Higher priority queues run first Example Timeline:

The Problem:

Once classified, processes are stuck forever!

What if P4 becomes interactive?

Batch queue may starve!

  • Time 0-2: System queue (P1) runs
  • Time 2-10: Interactive queue (P2, P3) run with RR
  • Time 10+: Batch queue (P4) finally runs
Critical Issue

This scenario requires careful synchronization. Uncontrolled concurrent access will violate mutual exclusion, leading to race conditions, data corruption, or deadlocks.

3.4 THE BIG PROBLEM

How Do We Know What a Process Needs?

The Classification Challenge:

When firefox.exe starts, is it:

What if processes could PROVE what they need by their BEHAVIOR?

This leads us to the most brilliant scheduling innovation...

WE CAN'T PREDICT BEHAVIOR JUST FROM THE NAME!

  • Downloading files? (I/O-bound)
  • Rendering video? (CPU-bound)
  • Sitting idle in background?

3.5 MLFQ - THE ADAPTIVE GENIUS

The Revolutionary Idea:

Start ALL processes at highest priority

If you use your full quantum → demoted (CPU-bound behavior)

If you yield before quantum → stay high (I/O-bound behavior)

Processes move dynamically based on actual behavior The Self-Organizing System:

CPU-bound processes naturally sink to lower queues I/O-bound processes naturally float to top This is brilliant! The system learns automatically!

 NO need to predict - let behavior decide!

3.6 MLFQ STRUCTURE

Notice:

  • Quantum increases as you go down
  • CPU-bound gets longer uninterrupted time
  • Interactive gets frequent turns at top

4.5 The Completely Fair Scheduler

The Philosophy Shift: What if we track FAIRNESS instead of queues?

Key Concept - Virtual Runtime (vruntime) Every process tracks: vruntime (virtual runtime) Scheduler rule: Always run process with LOWEST vruntime Example:

The Magic: Over time, everyone gets EXACTLY fair CPU share!

  • P1 runs for 10ms → vruntime += 10
  • P2 runs for 10ms → vruntime += 10
  • P1 runs again → vruntime += 10 (now 20)
  • Next: Schedule P2 (vruntime = 10 < 20)

4.6 CFS - THE NICE SYSTEM

Nice Values Control CPU Share:

Range: -20 (highest priority) to +19 (lowest) Default: 0 The Math:

Lower nice value → Higher weight → vruntime increases SLOWER Example:

P1: nice=0, weight=1024 P2: nice=5, weight=335 Both run 10ms of real time:

Result: P1 gets scheduled 3× more often!

  • P1: vruntime += 10 × (1024/1024) = 10
  • P2: vruntime += 10 × (1024/335) ≈ 30

Multi-Processor Scheduling

5.1 CFS Understanding Check

Linux CFS aims for:

A) Strict priority enforcement B) Fair CPU share based on weights C) Shortest job first optimization D) Round robin only ANSWER: B) Fair CPU share based on weights CFS ensures each process gets its fair share of CPU time proportional to its weight (determined by nice values).

5.2 WINDOWS PRIORITY SCHEDULING

Priority Structure:

Level 31-16: Real-time (fixed, never change) Level 15-1: Variable priority (dynamic) Level 0: System idle (reserved) Dynamic Adjustments:

I/O completion → Temporary priority boost

Foreground window → Priority boost

CPU-bound → Gradual priority decay back to base Philosophy: Responsiveness for user-facing tasks, throughput for background Windows uses explicit priorities with dynamic adjustments!

5.3 MULTI-PROCESSOR REVOLUTION

The New Challenge:

New Questions:

1. How to distribute load across CPUs evenly?

2. Should a process stick to one CPU?

3. What about shared caches?

4. How to synchronize schedulers?

The Opportunity: True parallelism!

4 CPUs can run 4 processes SIMULTANEOUSLY Single CPU: Who runs next?

Multi-CPU: Who runs WHERE?

5.4 LOAD BALANCING

Two Strategies:

Push Migration: Overloaded CPU pushes tasks away Pull Migration: Idle CPU pulls tasks from busy ones When to Balance?

  • Periodic check (every 100ms)
  • On idle detection
  • On new process creation

5.5 CPU AFFINITY - THE CACHE STORY

The Cache Scenario:

P1 runs on CPU 0:

P1 migrated to CPU 1:

The Solution - CPU Affinity:

Soft affinity: Prefer keeping process on same CPU Hard affinity: Lock process to specific CPU(s) Trade-off: Load balancing vs cache performance

  • Loads 10 MB of data into CPU 0's cache
  • Runs efficiently (cache hits)
  • CPU 1's cache is COLD (doesn't have P1's data)
  • Must reload 10 MB from RAM
  • Performance penalty: 100× slower initially!

5.6 Multi-Processor Check

In multi-processor systems, CPU affinity reduces:

A) Cache misses by keeping process on same CPU B) Load balancing effectiveness C) Preemption frequency D) Starvation risk ANSWER: A) Cache misses by keeping process on same CPU Affinity keeps the CPU cache "warm" with the process's data, avoiding the performance penalty of reloading data from main memory.

Trade-Offs & Key Takeaways

6.1 REAL-WORLD SCHEDULER SELECTION

System Type → Best Scheduler:

Web Server:

Real-Time System (Missile Guidance):

  • Workload: Thousands of short requests
  • Choice: MLFQ or CFS
  • Reason: Fair distribution, good response time
  • Workload: Few long-running simulations
  • Choice: SJF/SRTF or FCFS
  • Reason: Maximize throughput, predictability
  • Workload: Mix of everything
  • Choice: MLFQ or CFS with priority boosts
  • Reason: Responsiveness for user applications
  • Workload: Critical timing requirements
  • Choice: Priority scheduling with guarantees
  • Reason: Deterministic response times

6.2 Best Algorithm Check

What is the best algorithm for hard real-time systems?

A) FCFS B) Priority scheduling with guarantees C) Round Robin D) SJF ANSWER: B) Priority scheduling with guarantees Real-time systems need predictable, priority-based scheduling with guaranteed timing constraints.

Missing a deadline could be catastrophic.

6.3 THE TRADE-OFF TRIANGLE

The Reality: Every scheduler makes trade-offs!

  • FCFS: Simple + Fast, but unfair
  • RR: Simple + Fair, but not fastest
  • SRTF: Fair* + Fast, but complex
  • MLFQ: Balanced on all three

6.4 Overhead Understanding

Which algorithm has the HIGHEST overhead?

A) FCFS B) SRTF C) Non-preemptive priority D) Multi-level queues ANSWER: B) SRTF (frequent preemption) SRTF requires frequent context switches as shorter jobs arrive, causing high overhead from saving/restoring process state.

6.6 Key Takeaways
