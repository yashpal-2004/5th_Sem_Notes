1.3 Review - Why Paging?

Problems with Contiguous Allocation

Paging: The Solution

Why Paging Works

Key Advantage Logical and physical memory are separated enabling modern virtual memory systems.

  • External fragmentation wastes memory
  • Large processes need contiguous space
  • Compaction is costly and disruptive
  • Physical memory → fixed-size frames (e.g., 4KB)
  • Logical memory → fixed-size pages
  • Any page → any free frame
  • No external fragmentation
  • Simple, flexible allocation
  • Supports non-contiguous memory
  • Efficient memory utilization

1.4 Review - Address Translation in Paging

Virtual → Physical Address Virtual Address Format:

[ Page Number (p) | Offset (d) ]

Translation Steps

  1. CPU generates virtual address (p, d)
  2. p indexes the Page Table
  3. Page Table gives Frame number (f)
  4. Physical Address = (f, d)

Example Virtual Address = 10876 If Page 2 → Frame 5

Key Takeaway Paging translates addresses by table lookup + offset, enabling flexible, non-contiguous memory allocation.

  • Page size = 4KB (4096 bytes)
  • Offset bits = 12
  • Page = 10876 / 4096 = 2
  • Offset = 10876 % 4096 = 2684
  • Physical Address = (5 × 4096) + 2684 = 23164

1.6 The Performance Problem

Why Paging Seems Slow For every memory access, paging needs:

Two memory accesses per instruction

Example Effective Access Time: 100 ns (page table) + 100 ns (data) = 200 ns

100% slowdown

Impact

Conclusion Paging needs a faster address translation mechanism to be practical.

  • 1⃣ Access Page Table
  • 2⃣ Access Actual Data
  • Memory access time = 100 ns
  • High latency
  • Poor CPU utilization
  • Unacceptable for modern systems

Translation Lookaside Buffer (TLB)

2.1 Translation Lookaside Buffer (TLB)

What Is a TLB?

Key Features

Why TLB Matters

  • A small, very fast cache for page table entries
  • Also called Address Translation Cache
  • Located inside the MMU
  • Stores recent page → frame mappings
  • Fully / Set associative (parallel search)
  • 1 CPU cycle access (< 1 ns)
  • Small size (~64–256 entries)
  • Avoids frequent page table memory access
  • Reduces address translation from 2 accesses → 1
  • Dramatically improves paging performance

2.2 Translation Lookaside Buffer (TLB)

TLB Entry (What It Stores)

Purpose Speed up address translation by caching the most-used page table entries.

  • Valid bit – entry usable or not
  • Tag (Page #) – virtual page number
  • Frame # – physical frame
  • Protection bits – R / W / X
  • Dirty bit – page modified?
  • Reference bit – recently used?

2.3 TLB Hit and Miss Scenarios

Scenario 1: TLB Hit (Fast Path)

Time: ~1 ns

Key Takeaway System performance depends heavily on TLB hit rate—hits are fast, misses are expensive.

Scenario 2: TLB Miss (Slow Path) No TLB match

Time: ~100+ ns (first access)

  • CPU generates virtual address (Page P, Offset)
  • TLB searched in parallel
  • Match found → Frame F
  • Physical Address = (F, Offset)
  • Memory accessed directly
  • CPU generates virtual address (Page P, Offset)
  • Access Page Table in memory (via PTBR)
  • Get Frame F
  • Update TLB (may replace old entry)
  • Access memory with physical address

2.4 TLB Performance Metrics

Key Terms

Effective Access Time (EAT) General Formula:

EAT = ε + α·m + (1 − α)·(m + m)

What It Means

Example Insight

Key Takeaway High TLB hit ratio is crucial—it keeps paging performance close to raw memory speed.

  • Hit Ratio (α): Fraction of accesses found in TLB
  • Miss Ratio: 1 − α
  • TLB Access Time (ε): ~1 ns
  • Memory Access Time (m): ~100 ns
  • TLB Hit: 1 memory access
  • TLB Miss: 2 memory accesses (page table + data)
  • Higher α → EAT approaches m
  • α = 0.99 → EAT ≈ 1.01·m (near optimal)
  • Small drops in α cause big slowdowns

2.5 TLB Performance Example

Given: TLB access (ε) ≈ 0, Memory access (m) = 100 ns and Hit ratio (α) = 0.98

EAT Calculation

EAT = α·m + (1 − α)·2m
= 0.98·100 + 0.02·200 = 98 + 4 = 102 ns

Performance Impact

Hit Ratio Sensitivity

Key Takeaway Even a small drop in TLB hit ratio causes a big performance hit—high α is critical.

  • Without TLB: 200 ns
  • With TLB (98%): 102 ns
  • Speedup: ~1.96×
  • 99% → 101 ns
  • 95% → 105 ns
  • 80% → 120 ns

TLB Hit & Miss Performance

3.1 Why TLB Works - Locality of Reference

Core Idea TLB is effective because programs show locality of reference.

Spatial Locality

One TLB entry covers many accesses

Temporal Locality

Working Set

Example

Key Takeaway

High locality + small working set = TLB hits most of

the time, making paging fast.

  • Nearby memory locations accessed together
  • Examples: sequential code, array traversal
  • Many accesses fall within the same page
  • Recently used pages are reused soon
  • Examples: loops, frequently called functions
  • TLB keeps recent translations
  • Program uses a small active set of pages
  • Working set usually fits in the TLB
  • Typical hit ratios: 95–99%
  • Loop: array[0] … array[999]
  • Page size = 4KB ≈ 1024 integers
  • 1 TLB miss → 999 hits

3.2 TLB Management Issues

Context Switching Problem Using them → wrong physical memory access

Solution 1: Flush TLB Simple Cold start → many TLB misses

Solution 2: ASID (Address Space ID) No flush needed Faster context switches

  • Each process has its own page table
  • On context switch, TLB may contain old process entries
  • Invalidate all TLB entries on switch
  • New process starts with empty TLB
  • Tag each TLB entry with Process ID
  • Format: | ASID | Page # | Frame # |
  • Multiple processes coexist in TLB

3.3 TLB Replacement Policies

When TLB Is Full A new entry must replace an existing one — which to evict?

Common Policies

LRU (Least Recently Used)

NMRU (Not Most Recently Used)

FIFO (First In First Out)

Random

Key Takeaway Hardware favors simple, fast policies—Random and NMRU are common, while true LRU is rare.

  • Best performance (uses locality)
  • High hardware cost
  • Near-LRU performance
  • Lower complexity than LRU
  • May remove frequently used entries
  • Evict a random entry
  • Very simple, low overhead

Key Takeaway Polling is simple, interrupts are efficient, DMA is optimal—modern systems rely heavily on interrupts and DMA.

3.6 Polling (Programmed I/O)

What It Is CPU continuously checks device status until it’s ready.

How It Works

  1. CPU sends command to device
  2. Busy-wait loop checks status
  3. Device becomes ready
  4. CPU reads data and processes it

Simple Example // busy wait data = read_keyboard();

Characteristics Wastes CPU cycles Poor efficiency

  • Very simple
  • No special hardware
while (keyboard_status != READY) {

4.1 Polling - Advantages and Disadvantages

When to Use Polling

  • High-speed devices (negligible wait)
  • Real-time systems with strict timing
  • Small embedded systems (no interrupts)
  • Acceptable if combined with sleep/yield

4.2 Interrupt-Driven I/O

Device Signals CPU When Ready

How It Works

  1. CPU starts I/O ○ Sends command to device ○ Continues executing other tasks
  2. Device works independently ○ Performs I/O operation
  3. Interrupt raised ○ Device signals CPU on completion
  4. CPU handles interrupt ○ Saves context (PC, registers) ○ Runs ISR to read data ○ Restores context and resumes work

Key Advantage

  • No busy waiting
  • CPU remains productive while I/O runs.

4.3 Interrupt Processing Steps

How an Interrupt Is Handled (Step-by-Step)

  1. Device Ready ○ Device controller raises an interrupt signal
  2. Interrupt Detection ○ CPU detects interrupt at end of instruction cycle
  3. Interrupt Acknowledgment ○ CPU sends ACK ○ Device places interrupt number on bus
  4. Context Save ○ Save PC, registers, flags ○ Switch to kernel mode
  5. Interrupt Vector Lookup ○ Use interrupt number to index Interrupt Vector Table (IVT) ○ Fetch ISR address
  6. Execute ISR ○ Run device-specific handler ○ Read data / update buffers
  7. Context Restore ○ Restore saved state ○ Resume interrupted process

4.4 Interrupt Vector Table

What is IVT?

A table that maps interrupt numbers to their Interrupt Service Routine (ISR) addresses.

How It Works

  1. Device raises interrupt with interrupt number
  2. CPU uses number as index into IVT
  3. Fetches ISR address
  4. Direct jump to correct handler

IVT Structure

Key Points

  • One entry per interrupt
  • Enables fast ISR dispatch
  • Set up by OS at boot time
  • Core to interrupt-driven I/O

4.5 Interrupt Types and Priorities

By Source Hardware Interrupts Software Interrupts Exceptions / Traps

By Priority Non-Maskable Interrupts (NMI) Maskable Interrupts When multiple interrupts occur:

Higher priority is handled first.

  • From I/O devices (keyboard, disk, timer)
  • Asynchronous
  • Signal external events
  • Triggered by instructions (e.g., system calls)
  • Synchronous
  • OS service requests
  • Program-generated errors
  • Divide by zero, page fault, breakpoints
  • Cannot be disabled
  • Critical events (hardware failure, power loss)
  • Can be enabled/disabled
  • Normal I/O devices

I/O Management (Polling vs. Interrupts)

5.1 Problem with Interrupt-Driven I/O

The Data Transfer Bottleneck

Scenario: Read 1MB from Disk interrupts

Why This Is a Problem

Key Insight Interrupt-driven I/O works well for small or infrequent data, but becomes inefficient for large transfers.

  • Disk transfers 4 bytes at a time
  • Each transfer triggers an interrupt
  • Total interrupts: 1MB / 4 bytes = 262,144
  • CPU involved in every tiny transfer
  • Massive context switching
  • Wasted memory bandwidth
  • CPU can’t do useful work
  • Poor scalability for large I/O

5.2 Direct Memory Access (DMA)

What is DMA?

○ Start transfer ○ Completion interrupt

How DMA Works

  1. CPU sets up DMA (buffer address, size, direction)
  2. DMA controller takes control of the system bus
  3. Data moves Device ⇄ Memory directly
  4. DMA raises an interrupt when done

Key Idea

One-Line Takeaway

DMA = High-speed data transfer with minimal CPU involvement
  • Uses a DMA Controller (special hardware)
  • Transfers data directly between device and memory
  • CPU involved only twice:
  • DMA controller acts as a bus master
  • CPU is free to do other work during transfer

5.3 DMA Operation Steps

How a DMA Transfer Works 1⃣ CPU Setup ○ Source address ○ Destination address ○ Transfer size ○ Mode (read/write) CPU resumes other work 2⃣ DMA Takes Control DMA becomes bus master 3⃣ Data Transfer No CPU involvement 4⃣ Completion Interrupt sent to CPU

  • CPU programs DMA with:
  • DMA requests system bus
  • CPU grants bus
  • DMA moves data Device ⇄ Memory
  • Repeats until count = 0
  • DMA releases bus
  • CPU handles completion in ISR

5.4 DMA Transfer Modes

5.5 DMA vs CPU Transfer - Performance

Scenario: Transfer 1MB from disk → memory

1. CPU Programmed I/O (Polling) ~500 ms CPU usage: ~100% Extremely inefficient

2. Interrupt-Driven I/O ~2 μs per interrupt ~500 ms interrupt overhead CPU partially blocked by interrupts

3. Direct Memory Access (DMA) CPU involvement: ~12 μs total CPU ~99.9% free

Bottom Line

  • CPU moves 4 bytes at a time
  • 262,144 iterations
  • 262,144 interrupts
  • CPU setup: ~10 μs
  • DMA transfer: ~10 ms (100 MB/s)
  • Completion interrupt: ~2 μs
  • DMA reduces CPU overhead by ~40,000×
  • Best choice for large data transfers
  • Polling & interrupts don’t scale

5.6 DMA Advantages and Disadvantages

Best Use Cases

Key Takeaway:

DMA = maximum performance for large data, minimal CPU cost.

  • Disk I/O, network packets
  • Video/audio streaming
  • Bulk transfers > 1KB
  • High-bandwidth peripherals

Explanation:

DMA transfers data without CPU involvement, making it vastly more efficient than interrupt-based transfers.

(A) Interrupt-Driven I/O Interrupt frequency:

1 interrupt per 512 bytes Total interrupts:

Interrupts=10,485,760/512=20,480

6.2 Cache Coherency Problem with DMA

Keeping CPU Cache & Main Memory Consistent

The Problem DMA accesses main memory directly, bypassing CPU cache.

DMA Write (Device → Memory) CPU reads stale cached data DMA Read (Memory → Device) DMA reads stale memory data

Solutions Flush cache before DMA read Invalidate cache after DMA write Use uncached memory for DMA buffers Hardware cache coherence

Key Takeaway DMA is fast but unsafe without cache management.

Proper cache control is essential for correct and consistent data.

  • DMA updates memory
  • CPU cache still holds old value
  • CPU updates cache only
  • Memory not updated yet

6.3 Comparison Summary - Polling vs Interrupts vs DMA

Key Takeaway As data size and device speed increase, move from Polling → Interrupts → DMA for better performance and CPU efficiency.
