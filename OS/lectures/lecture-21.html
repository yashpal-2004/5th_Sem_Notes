1.3 Review - Why Paging?

02

❌

Problems with Contiguous Allocation

✅

Paging: The Solution

🚀

Why Paging Works

🔑

Key Advantage

Logical and physical memory are separated

enabling modern virtual memory systems.

  • External fragmentation wastes memory
  • Large processes need contiguous space
  • Compaction is costly and disruptive
  • Physical memory → fixed-size frames (e.g., 4KB)
  • Logical memory → fixed-size pages
  • Any page → any free frame
  • No external fragmentation
  • Simple, flexible allocation
  • Supports non-contiguous memory
  • Efficient memory utilization

1.4 Review - Address Translation in Paging

03

🧠

Virtual → Physical Address

Virtual Address Format:

[ Page Number (p) | Offset (d) ]

🔄

Translation Steps

1. CPU generates virtual address (p, d)

2. p indexes the Page Table

3. Page Table gives Frame number (f)

4. Physical Address = (f, d)

📊

Example

Virtual Address = 10876

If Page 2 → Frame 5

🔑

Key Takeaway

Paging translates addresses by table lookup + offset,

enabling flexible, non-contiguous memory allocation.

  • Page size = 4KB (4096 bytes)
  • Offset bits = 12
  • Page = 10876 / 4096 = 2
  • Offset = 10876 % 4096 = 2684
  • Physical Address = (5 × 4096) + 2684 = 23164

Translation Lookaside Buffer (TLB)

2.1 Translation Lookaside Buffer (TLB)

06

🧠

What Is a TLB?

⚙

Key Features

🚀

Why TLB Matters

  • A small, very fast cache for page table entries
  • Also called Address Translation Cache
  • Located inside the MMU
  • Stores recent page → frame mappings
  • Fully / Set associative (parallel search)
  • 1 CPU cycle access (< 1 ns)
  • Small size (~64–256 entries)
  • Avoids frequent page table memory access
  • Reduces address translation from 2 accesses → 1
  • Dramatically improves paging performance

2.2 Translation Lookaside Buffer (TLB)

07

🧩

TLB Entry (What It Stores)

🎯

Purpose

Speed up address translation by caching the

most-used page table entries.

  • Valid bit – entry usable or not
  • Tag (Page #) – virtual page number
  • Frame # – physical frame
  • Protection bits – R / W / X
  • Dirty bit – page modified?
  • Reference bit – recently used?

08

2.4 TLB Performance Metrics

09

📊

Key Terms

🧮

Effective Access Time (EAT)

General Formula:

EAT = ε + α·m + (1 − α)·(m + m)

⚡

What It Means

📈

Example Insight

🔑

Key Takeaway

High TLB hit ratio is crucial—it keeps paging

performance close to raw memory speed.

  • Hit Ratio (α): Fraction of accesses found in TLB
  • Miss Ratio: 1 − α
  • TLB Access Time (ε): ~1 ns
  • Memory Access Time (m): ~100 ns
  • TLB Hit: 1 memory access
  • TLB Miss: 2 memory accesses (page table + data)
  • Higher α → EAT approaches m
  • α = 0.99 → EAT ≈ 1.01·m (near optimal)
  • Small drops in α cause big slowdowns

2.5 TLB Performance Example

10

Given: TLB access (ε) ≈ 0, Memory access (m) = 100 ns

and Hit ratio (α) = 0.98

🧮

EAT Calculation

EAT = α·m + (1 − α)·2m
= 0.98·100 + 0.02·200 = 98 + 4 = 102 ns

⚡

Performance Impact

📈

Hit Ratio Sensitivity

🔑

Key Takeaway

Even a small drop in TLB hit ratio causes

a big performance hit—high α is critical.

  • Without TLB: 200 ns
  • With TLB (98%): 102 ns
  • Speedup: ~1.96×
  • 99% → 101 ns
  • 95% → 105 ns
  • 80% → 120 ns

3.1 Why TLB Works - Locality of Reference

12

🧠

Core Idea

TLB is effective because programs show locality of

reference.

📍

Spatial Locality

➡

One TLB entry covers many accesses

⏱

Temporal Locality

🧩

Working Set

📊

Example

🔑

Key Takeaway

High locality + small working set = TLB hits most of

the time, making paging fast.

  • Nearby memory locations accessed together
  • Examples: sequential code, array traversal
  • Many accesses fall within the same page
  • Recently used pages are reused soon
  • Examples: loops, frequently called functions
  • TLB keeps recent translations
  • Program uses a small active set of pages
  • Working set usually fits in the TLB
  • Typical hit ratios: 95–99%
  • Loop: array[0] … array[999]
  • Page size = 4KB ≈ 1024 integers
  • 1 TLB miss → 999 hits

4.2 Interrupt-Driven I/O

19

🔔

Device Signals CPU When Ready

⚙

How It Works

1. CPU starts I/O

○ Sends command to device

○ Continues executing other tasks

2. Device works independently

○ Performs I/O operation

3. Interrupt raised

○ Device signals CPU on completion

4. CPU handles interrupt

○ Saves context (PC, registers)

○ Runs ISR to read data

○ Restores context and resumes work

✅

Key Advantage

  • No busy waiting
  • CPU remains productive while I/O runs.

4.3 Interrupt Processing Steps

20

🧭

How an Interrupt Is Handled (Step-by-Step)

1. Device Ready

○ Device controller raises an interrupt signal

2. Interrupt Detection

○ CPU detects interrupt at end of instruction cycle

3. Interrupt Acknowledgment

○ CPU sends ACK

○ Device places interrupt number on bus

4. Context Save

○ Save PC, registers, flags

○ Switch to kernel mode

5. Interrupt Vector Lookup

○ Use interrupt number to index Interrupt Vector

Table (IVT)

○ Fetch ISR address

6. Execute ISR

○ Run device-specific handler

○ Read data / update buffers

7. Context Restore

○ Restore saved state

○ Resume interrupted process

5.1 Problem with Interrupt-Driven I/O

24

🚧

The Data Transfer Bottleneck

📌

Scenario: Read 1MB from Disk

interrupts

❌

Why This Is a Problem

🔑

Key Insight

Interrupt-driven I/O works well for small or infrequent

data, but becomes inefficient for large transfers.

  • Disk transfers 4 bytes at a time
  • Each transfer triggers an interrupt
  • Total interrupts: 1MB / 4 bytes = 262,144
  • CPU involved in every tiny transfer
  • Massive context switching
  • Wasted memory bandwidth
  • CPU can’t do useful work
  • Poor scalability for large I/O

How DMA Works

1. CPU sets up DMA (buffer address, size, direction)

Key Idea

DMA = High-speed data transfer with minimal CPU involvement

  • DMA controller acts as a bus master
  • CPU is free to do other work during transfer

6.2 Cache Coherency Problem with DMA

31

Keeping CPU Cache & Main Memory Consistent

⚠

The Problem

DMA accesses main memory directly, bypassing CPU cache.

DMA Write (Device → Memory)

CPU reads stale cached data

DMA Read (Memory → Device)

DMA reads stale memory data

🛠

Solutions

Flush cache before DMA read

Invalidate cache after DMA write

Use uncached memory for DMA buffers

Hardware cache coherence

✅

Key Takeaway

DMA is fast but unsafe without cache management.

Proper cache control is essential for correct and consistent data.

  • DMA updates memory
  • CPU cache still holds old value
  • ❌
  • CPU updates cache only
  • Memory not updated yet
