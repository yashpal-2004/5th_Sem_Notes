Concurrency vs. Parallelism

1.1 Core Difference

Concurrency is about dealing with lots of things at once (structure). Parallelism is about doing lots of things at once (execution).

1.2 The Coffee Shop Analogy

Coffee Shop Analogy
  • Concurrency (Single Core): One barista, two lines of customers. The barista takes an order from Line A, starts the machine, switches to Line B, takes payment, and switches back to A. They are making progress on both lines, but only working on one at a time.
  • Parallelism (Multi-Core): Two baristas, two machines. Two customers get served at the exact same instant.
2

Multithreading Models

2.1 User-Level Threads (ULT)

Also known as "Invisible Threads", "Green Threads", or N:1 threads. The OS is oblivious and sees only one single process. A user-space library (like the JVM) slices time and switches between tasks manually.

  • Pros: Context switches are extremely fast (nanoseconds) because no system calls are needed.
  • Cons (The Fatal Flaw): If one thread calls a blocking operation (like reading a file), the OS pauses the entire process because it doesn't know there are other threads. The whole app freezes.

2.2 Kernel-Level Threads (KLT)

Also known as Native Threads or 1:1 threads. The OS is fully aware and allocates a kernel structure for every thread.

  • Pros: True parallelism. If one thread blocks on I/O, the OS simply switches to another thread.
  • Cons: Expensive context switches (microseconds) requiring mode switches between user and kernel space.

2.3 Hybrid Threads (M:N)

A pool of user threads is multiplexed onto a smaller pool of kernel threads.

Ride-Sharing Analogy

Passengers = User Threads. Cars = Kernel Threads. A car picks up a passenger, drives them. If the passenger needs to stop (blocking I/O), they get out. The car immediately picks up the next passenger. The car never waits.

Pros: High concurrency (millions of threads) with true parallelism.

Cons: Extremely complex to implement.

2.4 Virtual Threads (Java 21+)

A modern, production-ready implementation of the M:N model.

  • Mounting: JVM assigns a Virtual Thread to a "Carrier Thread" (Kernel Thread).
  • Unmounting: When the code hits a blocking call, the JVM detaches it and stores it in heap memory, while the Carrier Thread grabs a new Virtual Thread to run.

Allows for creating 10,000,000 threads on a standard laptop!

3

The Evolution of Execution

3.1 Phase 1: Sequential Execution

Program runs strictly top-to-bottom. If a download takes 5 seconds, the CPU sits idle for 5 seconds waiting. Very inefficient.

3.2 Phase 2: Cooperative Multitasking (User-Level)

Tasks voluntarily "yield" control. Efficient, but dangerous.

Danger Scenario

If Task A gets into an infinite loop and forgets to yield, Task B starves. One rogue task ruins the entire system.

3.3 Phase 3: Preemptive Multitasking (Kernel-Level)

The OS scheduler handles context switching automatically. True parallelism. The OS is the boss, not the code. Eliminates the requirement for manual yielding.

4

OS Implementations

4.1 Linux & Java Realities

The Linux Revelation: In Linux, there is no separate concept of a "thread". It uses task_struct. Whether it acts like a process or thread depends purely on the sharing flags passed to the clone() system call.

Java's Evolution:

  • JDK 1.0: Green Threads (N:1).
  • JDK 1.2 - 19: Native Threads (1:1). Stable, but heavy.
  • JDK 21+: Virtual Threads (M:N). Returned to N:1 concepts with modern hardware support.
5

Hazards & Synchronization

With great power comes great synchronization problems.

5.1 Race Conditions

Even a simple count++ is actually three instructions: Read, Add, Write.

The "Lost Update"

If two threads try to increment 5 at the exact same time, they might both read 5, both add 1, and both write 6 back to memory. The count should be 7, but an update was lost.

5.2 Deadlocks

Analogy: Four cars at a 4-way stop with the rule "Yield to the right". If 4 cars arrive simultaneously, everyone waits for the person on their right. No one moves. Forever.

The Technical Fix: Lock Ordering. Ensure threads always acquire locks in the exact same predefined order (e.g., always acquire Lock A before Lock B).

Pthread API

6.1 Core Functions

Pthreads (POSIX Threads) is the standard C/C++ API for thread management.

// Create a thread
int pthread_create(pthread_t *thread, const pthread_attr_t *attr, void *(*start_routine)(void*), void *arg);

// Wait for a thread (like waitpid)
int pthread_join(pthread_t thread, void **retval);

// Fire and forget (system reclaims resources automatically)
int pthread_detach(pthread_t thread);

// Terminate calling thread
void pthread_exit(void *retval);
Important

You must call pthread_join() on joinable threads to release their resources, otherwise they become zombie threads.
