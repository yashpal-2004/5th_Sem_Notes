POSIX Interface

1.1 What is POSIX?

POSIX (Portable Operating System Interface) is a set of standards (rulebook) created to make different operating systems behave in a similar, predictable way — especially Linux, Unix, macOS, and Solaris.

Did you know?

Apple's macOS (formerly Mac OS X) and iOS are UNIX-certified and POSIX compliant. This is because macOS uses the XNU kernel (hybrid kernel) which incorporates BSD (UNIX-based, POSIX compatible).

Before POSIX, every UNIX version was different. Programs written for one system often did not run on another.

POSIX standardizes:

  • fork() and exec()
  • open(), read(), write()
  • Threads (pthreads)
  • File permissions and shell commands
2

fork() System Call

2.1 Creating a Duplicate

fork() is a POSIX system call used to create a new process. It creates an almost exact copy of the current running program.

  • Same: code, environment, open file descriptors.
  • Same Layout: Code, Data, Stack (but separate physical memory).
  • Different: Allocates a new PID for the child.

Note: Each process continues executing independently from the exact next line after fork().

2.2 The Photocopy Analogy

Photocopy Machine

Imagine you (the parent process) give a page to a photocopy machine:

  • The machine creates a duplicate page (child process).
  • Both pages initially have the same content.
  • You can write on your page, and someone else can write on the duplicate. These actions do not affect each other.

2.3 Return Values

// C Example
#include <stdio.h>
#include <unistd.h>

int main() {
    pid_t id = fork();
    if (id == 0) {
        printf("This is Child Process, pid = %d\n", id);
    } else {
        printf("This is Parent Process. Child PID = %d\n", id);
    }
    return 0;
}
  • 0: You are inside the new child process.
  • >0 (positive): Parent process. Value = Child's PID.
  • -1: fork failed, no child created.
3

Memory Model & Copy-on-Write

3.1 COW Optimization

When fork() is called, the OS does not immediately copy the entire memory. This would be too slow.

It uses Copy-On-Write (COW):

  • Parent and child share memory pages initially. Pages are marked as read-only.
  • If either modifies a page, the OS intercepts the write and creates a separate copy just for that page.
  • This saves massive amounts of memory and time, especially since most child processes immediately call exec() anyway.

3.2 vfork() System Call

vfork() is like fork() but with NO copying or COW.

Danger!

The child shares the parent's exact stack, data segment, and heap. The parent process is completely suspended until the child calls exec() or _exit().

It is dangerous because the child can corrupt parent memory or cause stack clashing. macOS man pages describe it as "obsolete".

4

exec() System Call Family

4.1 Program Replacement

exec() replaces the current running program with a new program.

  • No new process is created: The PID stays exactly the same.
  • After a successful exec(), the old code is gone and cannot resume. The code segment, data, heap, and stack are replaced.
  • File descriptors and User/group IDs remain the same.

4.2 Syntax and Variants

There are several variants (execl, execlp, execv, execve, etc.):

  • l vs v: 'l' takes arguments as a list. 'v' takes arguments as a vector (array).
  • p: Searches the PATH for the executable.
  • e: Allows passing a custom environment array.
// execl(path, arg0, arg1, ..., NULL)
execl("/bin/ls", "ls", "-l", "-a", NULL);

Note: execve() is the actual system call used by the kernel; the others are C library wrappers.

5

Process Termination

5.1 Normal vs Abnormal

  • Normal Termination: Exits gracefully. Open files closed, buffers flushed to disk, exit handlers called. Caused by returning from main() or calling exit().
  • Abnormal Termination: Unexpected kill. No cleanup, buffers may be lost. Caused by signals (SIGKILL, SIGSEGV, Ctrl+C).

5.2 exit() and atexit()

exit(int status) is a C library function that performs cleanup:

  1. Calls functions registered with atexit() in reverse order (LIFO).
  2. Flushes all open stdio streams.
  3. Closes all open streams and removes temp files.
  4. Calls _exit() to enter the kernel.

5.3 _exit() System Call

_exit() is a direct system call that immediately terminates without any cleanup. No buffers flushed, no atexit() handlers called. It is highly recommended to use _exit() inside a child process after a fork() to prevent it from accidentally flushing the parent's buffers or calling the parent's atexit handlers!

6

wait() and waitpid()

6.1 Why we need wait()

The wait() family allows a parent to sleep until its child terminates. Why is this needed?

  • Synchronization: Wait for a child to complete a task.
  • Zombie Prevention: Reaps terminated children (cleans up their PCB).
  • Status Collection: Retrieve the child's exit code.

6.2 waitpid() Options

pid_t waitpid(pid_t pid, int *status, int options);
  • > 0: Wait for a specific child PID.
  • -1: Wait for ANY child (same behavior as basic wait()).
  • 0: Wait for any child in the same process group.
7

Zombie & Orphan Processes

7.1 Zombies

Zombie Process: A process that has finished executing but still exists in the system's process table.

It happens because a child exits, but the parent hasn't called wait() yet. The OS must keep a minimal PCB around so the parent can read the exit status later. Zombies consume no CPU or memory, but they do consume a PID slot. Calling wait() completely removes the zombie.

7.2 Orphans

Orphan Process: A process whose parent has died before the child finishes.

When this happens, the kernel steps in and changes the child's Parent PID (PPID) to 1. The child is adopted by init (or launchd on macOS/ systemd on Linux). The init process automatically calls wait() on its adopted children, ensuring they don't become zombies when they eventually finish.

8

Threads (Lightweight Processes)

8.1 Process vs Thread

A thread is a lightweight unit of execution within a process that shares resources but executes independently.

Analogy
  • Process: A house with its own address, utilities, and furniture.
  • Thread: Multiple people living in the same house, sharing resources (kitchen) but doing different tasks.

8.2 Memory Model & Benefits

Shared by Threads: Code, Data (globals), Heap, open files, PID.

Separate per Thread: Stack (local variables), Registers, Program Counter, Thread ID (TID).

Benefits of Multithreading
  • Performance: Creation and context-switching is much faster than full processes.
  • Easy Communication: They share the same memory space, so no complex IPC is needed.
  • Responsiveness: E.g., a Web Browser can have one thread downloading images while another renders the UI.
