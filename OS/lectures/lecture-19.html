1.3 Introduction to Memory Management

Definition Memory management is how the OS controls and optimizes main memory for running processes.

Core Responsibilities Track used vs free memory Allocate memory to processes Free memory when done Protect process memory Optimize performance

Why It Matters Physical memory is limited Multiple processes compete Need isolation & safety Faster memory = better performance Efficient use of available RAM

Key Challenge How can limited physical memory be shared efficiently and safely among many processes?

1.4 Objectives of Memory Management

Relocation Load programs anywhere in memory; support dynamic execution Protection Isolate processes; protect OS/kernel memory Sharing Controlled memory sharing (libraries, IPC) Logical Organization Support modules, separate compile & link Physical Organization Move data between RAM ↔ storage, transparent to user Efficiency Minimize waste, faster access, better CPU use Key Takeaway Memory management balances safety, flexibility, and performance while using limited RAM efficiently.

1.5 The Memory Hierarchy

Core Idea Memory is organized in layers based on speed, cost, and capacity

Faster = Smaller & Costlier

Slower = Larger & Cheaper

Key Takeaway Systems trade speed for size and cost—fast memory is small, slow memory is large.

Hierarchy (Top → Bottom) CPU Registers Bytes | < 1 ns | Fastest, most expensive Cache (L1 / L2 / L3) KB–MB | 1–10 ns | Very fast Main Memory (RAM) GB | 50–100 ns | Moderate speed & cost Secondary Storage (SSD/HDD) TB | µs–ms | Slow, cheap Tertiary Storage (Tape/Cloud) Very large | seconds+ | Slowest, cheapest

1.6 Memory Hierarchy Characteristics

Core Principles Locality of Reference ○ Temporal: recent data used again soon ○ Spatial: nearby data used next Caching Strategy ○ Keep frequently used data in fast memory ○ Move rarely used data to slower memory Hierarchical Access ○ CPU checks fastest level first ○ On miss → check next level ○ Bring data up the hierarchy

Performance Impact Key Takeaway

High locality + good caching = faster memory access and better performance.
  • Hit Rate: % found in fast memory
  • Effective Access Time: Hit Rate × Fast Access + Miss Rate × Slow Access

2.3 Address Binding

What is Address Binding?

Mapping program addresses to actual memory locations Execution-Time (Dynamic) ○ Bound during execution ○ Process can move in memory ○ Maximum flexibility | Translation overhead ○ Used in modern OS Load-Time ○ Relocatable code ○ Bound when loaded ○ More flexible | Can’t move while running

Types of Binding Compile-Time ○ Fixed (absolute) addresses ○ Load at one location only ○ Fast | No flexibility ○ Example: old DOS programs

2.4 Address Binding

Address Translation Logical Address + Relocation Register → Physical Address Handled by MMU (hardware) Key Takeaway Later binding = more flexibility; modern systems use execution-time binding for safety and efficiency.

2.5 Protection in Contiguous Allocation

Why Protection Is Needed

Hardware Support: Base & Limit Registers Check on every access:

Physical Address = Base + A

Example

  • Prevent process → process memory access
  • Protect OS from user programs
  • Catch invalid memory references
  • Ensure stability & security
  • Base Register: Start address of process
  • Limit Register: Size of process memory
  • If A < 0 or A ≥ Limit → TRAP
  • Else → Access allowed
  • Base = 300KB, Limit = 120KB
  • Valid range: 300–420KB

2.6 Protection in Contiguous Allocation

Privilege Levels ○ Access all memory ○ Can change base/limit ○ Restricted to its memory only

3.1 Contiguous Memory Allocation

Definition Each process occupies one continuous block of physical memory.

Core Idea

Advantages overhead

Disadvantages process size Key Takeaway Contiguous allocation is fast and simple, but wastes memory and lacks flexibility.

Simple Memory Layout

Two Approaches Fixed Partitioning Predefined, fixed-size partitions Dynamic Partitioning Partitions created based on process size

  • Process uses consecutive addresses
  • No gaps inside a process
  • Entire process loaded as one block
  • Simple design, Fast access and Low
  • Fragmentation, Poor flexibility and Limits
  • OS in resident memory
  • User processes placed one after another
  • Remaining space = free memory

3.2 Fixed Partitioning - Concept

Definition Memory is split into fixed-size partitions at system startup.

One process per partition

Key Characteristics

Variants Equal-Size Partitions Same size; simple; causes more waste Unequal-Size Partitions Different sizes; better fit for varied processes

  • Partition sizes decided before boot
  • Sizes do not change during execution
  • Process must fit entirely in a partition

3.3 Fixed Partitioning - Concept

Example (Equal Size: 5 MB)

Internal Fragmentation

Advantages

Disadvantages Key Takeaway Fixed partitioning is easy to manage but wastes memory due to internal fragmentation.

  • P1 (3 MB) → 2 MB wasted
  • P2 (1 MB) → 4 MB wasted
  • P3 (4 MB) → 1 MB wasted
  • Simple & predictable
  • Easy allocation
  • Internal fragmentation
  • Fixed process limit
  • Poor fit for large/variable processes

3.4 Fixed Partitioning - Placement Strategies

Best Practices Key Takeaway Placement choice trades flexibility for simplicity. Internal fragmentation is unavoidable in fixed partitioning.

  • Prefer unequal partitions
  • Match sizes to workload
  • Review & adjust over time

3.5 Dynamic Partitioning - Concept

Definition Memory partitions are created on demand, sized exactly to the process.

Key Features No internal fragmentation

How Memory Changes

  • No fixed partition sizes
  • Partition = process size
  • Created on load, removed on exit
  • Start: OS + large free block
  • Load processes → memory splits dynamically
  • Free space shrinks and shifts over time

3.6 Dynamic Partitioning - Concept

Advantages

Disadvantages Key Takeaway Dynamic partitioning fits processes perfectly but suffers from external fragmentation over time.

  • No internal waste
  • Flexible for varied process sizes
  • Better initial memory use
  • External fragmentation develops
  • More complex allocation logic
  • Overhead tracking free blocks

Contiguous Allocation & Fixed Partitions

4.1 First Fit Allocation Algorithm

Idea Allocate the first free block that is large enough for the process.

How It Works 1⃣ Scan free memory from the start 2⃣ Pick the first hole ≥ process size 3⃣ Allocate memory 4⃣ Split hole if needed 5⃣ Stop searching

Example Free Holes: 50KB | 200KB | 100KB | 300KB Process: 80KB

Chosen: 200KB (first fit)

Remaining hole: 120KB

4.2 First Fit Allocation Algorithm

Pros

Cons

Performance Key Takeaway First Fit is quick and simple, but may cause fragmentation over time.

  • Fast & simple
  • Low overhead
  • Works well in practice
  • Small holes accumulate at start
  • Not always optimal
  • Fragmentation near beginning
  • Time: O(n) (scan free list)

4.5 Best Fit Allocation Algorithm

Idea Allocate the smallest free block that can fit the process

Leaves minimum leftover space

How It Works 1⃣ Scan entire free list 2⃣ Choose smallest hole ≥ process size 3⃣ Allocate & split remainder

Example Free Holes: 50KB | 200KB | 100KB | 300KB Process: 80KB

Chosen: 100KB (waste = 20KB)

4.6 Best Fit Allocation Algorithm

Pros

Cons

Performance Key Takeaway Best Fit reduces immediate waste but increases fragmentation and overhead over time.

  • Minimizes waste per allocation
  • Saves large holes for big processes
  • Good utilization (in theory)
  • Slower (full scan)
  • Creates many tiny holes
  • Often worse than First Fit in practice
  • Time: O(n) (checks all holes)

Fragmentation, Compaction & Coalescing

6.1 Memory Fragmentation

Definition Fragmentation = wasted memory that cannot be effectively used.

1. Internal Fragmentation Example:

100KB partition − 70KB process = 30KB wasted Formula:

Internal = Partition Size − Process Size

2. External Fragmentation Key Problem:

Enough total free memory, but no single large block

  • Waste inside allocated memory
  • Happens in Fixed Partitioning
  • Allocated > Needed
  • Waste between processes
  • Happens in Dynamic Partitioning
  • Free memory exists but not contiguous
Critical Issue

This scenario requires careful synchronization. Uncontrolled concurrent access will violate mutual exclusion, leading to race conditions, data corruption, or deadlocks.

6.2 Solutions to Fragmentation - Compaction

What It Is Rearrange memory so all free space becomes one large block.

How It Works

Effect

Requirements Processes movable at runtime

  • Move all processes to one end
  • Merge scattered holes
  • Update process addresses
  • Before: Many small holes (fragmented)
  • After: One big free block (contiguous)
  • Dynamic relocation
  • Hardware support (MMU)

6.3 Solutions to Fragmentation - Compaction

Pros

Cons

When to Use Key Takeaway Compaction fixes fragmentation but is costly—used sparingly in real systems.

  • Eliminates external fragmentation
  • Improves memory utilization
  • Enables large allocations
  • Very expensive (memory copying)
  • System pause required
  • Address updates needed
  • Time ≈ O(memory size)
  • Allocation fails despite enough total free memory
  • Low system activity
  • Fragmentation crosses threshold

6.4 Solutions to Fragmentation - Coalescing

What Is Coalescing?

Automatically merge adjacent free blocks when memory is released.

How It Works ○ Mark block as free ○ Check left & right neighbors ○ Merge if they’re free

Effect Without coalescing: many small holes With coalescing: one larger hole

  • On process termination:
  • Update free list

6.5 Solutions to Fragmentation - Coalescing

Implementation Boundary Tags ○ Size/status at both ends ○ O(1) neighbor checks Doubly Linked Free List ○ Easy merge with neighbors Key Takeaway Coalescing keeps free memory blocks large by merging neighbors—cheap, fast, and essential for dynamic allocation.

Pros

Cons

  • Prevents fragmentation from worsening
  • Fast (O(1))
  • No process relocation
  • Automatic
  • Doesn’t remove existing fragmentation
  • Extra bookkeeping

Swapping & Overlays

7.1 Swapping in Memory Management

What is Swapping?

Moving entire processes between RAM and disk to free memory.

Why Swapping?

Key Points Key Takeaway Swapping extends memory capacity but is costly—used carefully in real systems.

How It Works

Cost Swap Time

= Process Size / Disk Transfer Rate
Example: 100MB ÷ 50MB/s → 2s out + 2s in = 4s
  • Increase multiprogramming
  • Free memory for high-priority processes
  • Needs backing store (disk)
  • Requires contiguous memory (with swapping)
  • Excessive swapping → thrashing
  • Swap Out: RAM → Disk
  • Swap In: Disk → RAM
  • Very expensive (disk I/O)
  • Much slower than context switch

7.2 Implementing Swapping

Core Requirements Backing Store: Fast disk space for swapped processes Dynamic Relocation: Process may return to different memory location Roll Out / Roll In:

○ Low priority → swapped out ○ High priority → swapped in

Swapping Policies When to Swap Out Which Process

What Affects Swap Time Process Size → larger = slower Disk Speed → SSD ≫ HDD Memory Load → may need extra swaps

  • Low free memory, Long I/O wait and Lower priority process
  • Lowest priority, Longest blocked and Largest memory user
