1.3 Introduction to Process Synchronization

What is Process Synchronization?

A mechanism that ensures multiple concurrent processes or threads can safely share resources and coordinate their execution without conflicts.

Why Do We Need It?

Real-World Scenario:

Expected balance: ₹4,000 Possible actual balance: ₹3,000 or ₹6,000. Without proper synchronization, chaos!

  • You check your bank balance: ₹5,000
  • You withdraw ₹2,000 at ATM
  • Your friend deposits ₹1,000 online

1.4 Concurrent vs Parallel Execution

Two Ways to Run Multiple Processes Concurrency (Single CPU) Process A: ████░░░░████░░░░ Process B: ░░░░████░░░░████ Time → Interleaving - taking turns rapidly Parallelism (Multi-CPU) CPU 1: ████████████████ CPU 2: ████████████████ Time → Truly simultaneous execution Both need synchronization!

2.1 The Critical Section Problem

What is a Critical Section?

A critical section is a segment of code where a process accesses shared resources (variables, files, data structures) that must not be accessed by more than one process at a time.

The Danger Zone // Entry Section - request permission // CRITICAL SECTION - danger zone!

[access shared data] // Exit Section - release permission // Remainder Section - safe zone

do {
} while (TRUE);

2.4 Race Condition - Classic Example (Bank Account)

For Deposit (Process A):

int balance = read_balance(); // reads 5000
balance = balance + 1000; // calculates 6000

write_balance(balance); // writes 6000 For Withdrawal (Process B):

int balance = read_balance(); // reads 5000
balance = balance - 2000; // calculates 3000

write_balance(balance); // writes 3000 Expected Final Balance: 4000 (5000 + 1000 - 2000) The Bank Account Problem:

Consider a shared bank account with balance = 5000 Two processes execute simultaneously:

  • Process A: Deposits 1000
  • Process B: Withdraws 2000
Critical Issue

This scenario requires careful synchronization. Uncontrolled concurrent access will violate mutual exclusion, leading to race conditions, data corruption, or deadlocks.

3.2 Race Condition - Producer-Consumer Example

Bounded Buffer Problem Setup:

Shared buffer with counter variable tracking number of items.

Consumer Code: while (counter == 0) ; // buffer empty, wait item = buffer[out];

out = (out + 1) % BUFFER_SIZE;

counter--; // CRITICAL OPERATION The Problem: counter++ and counter-- are NOT atomic operations!

Producer Code:

// produce item while (counter == BUFFER_SIZE) ; // buffer full, wait buffer[in] = item;

in = (in + 1) % BUFFER_SIZE;

counter++; // CRITICAL OPERATION

while ( true) {
}
while (true) {
}

3.3 Why counter++ is Dangerous

Looks Simple, Right? counter++; // Just one line!

But In Machine Code Assembly LOAD R1, counter // Step 1: Read ADD R1, R1, 1 // Step 2: Calculate STORE R1, counter // Step 3: Write 3 separate instructions = 3 points where interruption can occur!

The Scenario:

Initial: counter = 5 Thread 1: counter++ (wants: 6) Thread 2: counter++ (wants: 7)

Race Condition Execution T1: LOAD R1, counter (R1 = 5) T2: LOAD R2, counter (R2 = 5) T1: ADD R1, R1, 1 (R1 = 6) T2: ADD R2, R2, 1 (R2 = 6) T1: STORE R1, counter (counter = 6) T2: STORE R2, counter (counter = 6) Expected: 7 Actual: 6 One increment lost!

3.4 Requirements - Critical Section Solutions

Every Solution MUST Satisfy All Three:

1⃣ Mutual Exclusion > Only ONE process in critical section at a time 3⃣ Bounded Waiting > Everyone gets their turn eventually (no starvation) Memory Aid: MPB (Mutual, Progress, Bounded) 2⃣ Progress > If no one's inside, someone waiting MUST get in

Race Conditions & Examples

4.1 Requirement 1 - Mutual Exclusion

The Bathroom Rule Only ONE person inside at a time Process P1: [inside critical section] Process P2: [must wait] Process P3: [must wait] Why It Matters

What happens if this is violated?

Answer: Data corruption!

  • Prevents simultaneous access
  • Protects data integrity
  • Most fundamental requirement

4.2 Requirement 2 - Progress

No Artificial Delays Bad Scenario:

Critical Section: [EMPTY] Process A: Waiting for Process B Process B: In remainder section (not even interested!)

Violation! Selection postponed indefinitely The Rule

  • If Critical Section is free AND processes want to enter
  • Decision MUST be made in finite time
  • Only waiting processes participate in decision

4.3 Requirement 3 - Bounded Waiting

Fairness Guarantee Bad Scenario:

Process A: Requested entry 1,000 times Process B: Keeps entering first Process A: Still waiting... (STARVATION) The Rule There exists a limit on how many times others can enter before you Not Necessarily FIFO, but no infinite waiting!
