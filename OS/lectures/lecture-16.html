1.4 What is a Deadlock?

Definition A state where processes wait on each other indefinitely, so no progress is possible.

Simplified Blocked processes holding resources while waiting for others.

Key Characteristics Permanent block (no timeout) Circular dependency No self-recovery Affects all involved processes Formal View P₀ → P₁ → … → Pⱼ → P₀

  • Each process waits for another
  • Circular wait exists:

2.3 Deadlock Characterization

Definition A set of processes is in deadlock when each process waits for an event that only another process in the set can cause.

Key Characteristics At least 2 processes At least 2 resources Circular wait No independent progress

2.5 Condition 1 Mutual Exclusion

Definition Only one process can use a resource at a time. Others must wait until it’s released.

Why It Exists Code Example: Mutex (C / POSIX) pthread_mutex_t lock; pthread_mutex_lock(&lock); // Acquire access_shared_resource(); // Critical section pthread_mutex_unlock(&lock); // Release Examples

Shareable (No Mutual Exclusion) code

Non-Sharable (Needs Mutual Exclusion) records and Critical sections (mutex)

  • Non-shareable resources (e.g., printer)
  • Data consistency (file writes)
  • Hardware limits
  • Read-only files, Read-only memory and Reentrant
  • Printers, scanners, Tape drives, Writable database

3.2 Condition 2 Hold and Wait

Scenario

The Four Necessary Coffman Conditions

4.1 Condition 4 Circular Wait

Definition A closed chain of processes exists where each process waits for a resource held by the next.

Formal View P₀ → P₁ → … → Pⱼ → P₀ (Pᵢ waits for a resource held by Pⱼ) Example Scenario

P1 → P2 → P3 → P1

Key Insight Circular wait is the final condition that completes deadlock.

Breaking the cycle ⇒ deadlock prevention.

Deadlock-Prone Code // Process P1 acquire(R1); acquire(R2); // waits if P2 holds R2 // Process P2 acquire(R2); acquire(R1); // waits if P1 holds R1

  • P1 holds R1, waits for R2
  • P2 holds R2, waits for R3
  • P3 holds R3, waits for R1

4.2 Code Example - Three-Process Circular Wait

Three-Process Circular Wait Circular wait chain:

Cycle: P1 → R2 → P2 → R3 → P3 → R1 → P1

  • P1 holds R1, wants R2
  • P2 holds R2, wants R3
  • P3 holds R3, wants R1

4.3 Resource Allocation Graph (RAG) Introduction

Purpose A directed graph that visually represents resource allocation and deadlock situations.

Graph Components

Vertices Processes (P) – Circles

Resources (R) – Rectangles Edges

Request Edge: Pᵢ → Rⱼ ⬅ Assignment Edge: Rⱼ → Pᵢ

Key Idea Cycles in the graph indicate possible or actual deadlock.

  • ● Dots inside = resource instances
  • Process Pᵢ is waiting for Rⱼ
  • Resource Rⱼ allocated to Pᵢ
● P = {P₁, P₂, …, Pⱼ}
● R = {R₁, R₂, …, Rⱼ}

4.4 RAG Notation and Symbols

Symbols Edges Request: Pᵢ → Rⱼ (process waiting) Assignment: Rⱼ → Pᵢ (resource held) Example How to Read the Graph resource instances Important Notes

  • ○ Process: P₁, P₂, …
  • □ Resource: R₁, R₂, …
  • • Dots: resource instances
  • ⬅
  • P₁ → R₁ : P₁ requests R₁
  • R₂ → P₃ : R₂ allocated to P₃
  • Edge into process → resource held
  • Edge out of process → waiting for
  • Edges from resource → allocated
  • Edges to resource → waiting requests
  • Request Assignment when granted
  • Assignment removed on release
  • Graph updates dynamically
Important Note

Make sure to enforce ordering and handle edge conditions to ensure correctness and stability.

5.4 Deadlock Example Dining Philosophers

Problem Setup Deadlock Scenario 1. All pick up left chopstick 2. All try to pick up right chopstick 3. Right chopstick held by neighbor 4. Everyone waits forever

DEADLOCK Why Deadlock Occurs (All 4 Conditions) Mutual Exclusion: One chopstick → one philosopher Hold & Wait: Holding left, waiting for right No Preemption: Cannot force release Circular Wait: P₁ → P₂ → … → P₅ → P₁

Key Insight Dining Philosophers is a classic deadlock illustration showing how all four conditions can arise naturally.

  • 5 philosophers, circular table
  • 5 chopsticks (one between each pair)
  • Each philosopher needs 2 chopsticks
  • Alternates between thinking and eating

RAG Analysis Rules

6.1 Necessary vs Sufficient Conditions

All 4 Coffman Conditions are necessary:

Circular Wait

If any one is missing → NO deadlock Key Distinction

All 4 present ≠ guaranteed deadlock

All 4 + single instances + cycle = deadlock

Practical Insight

Prevent deadlock by breaking at least ONE necessary condition Sufficient Conditions Presence guarantees deadlock Deadlock is certain when:

  • Mutual Exclusion, Hold and Wait, No Preemption and
  • Necessary → “Must have”
  • Sufficient → “Guarantees”
  • All 4 conditions present
  • Each resource has one instance
  • Cycle exists in RAG
