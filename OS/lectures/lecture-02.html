The Central Mystery & Challenges

1.1 The Miracle

Every second, your computer performs a miracle. You double-click an app icon... and dead code on disk turns into a living process in memory.

  • How does the OS bring code to life?
  • How does it manage 100+ processes simultaneously?
  • How does it switch between them so fast you don't notice?

1.2 Challenge 1: Design

How do you build an OS that's both FAST and SAFE? This shapes everything that follows.

1.3 Challenge 2 & 3: Multiplicity & Management

Challenge 2: Multiplicity. How do you run 100 programs on 1 CPU? The illusion of parallelism.

Challenge 3: Management. How do you track and control everything happening? The OS needs perfect memory.

2

The Design Dilemma

You're building an OS. Where do you put everything?

2.1 Option A: All Together (Monolithic)

Everything in kernel space. Device drivers, file systems, networking—all in the kernel.

  • Why It's Awesome: No boundaries = No overhead. Direct function calls = Lightning speed. Simple communication between components.
  • Why It's Terrifying: A graphics driver bug? Kernel panic. One mistake = Blue screen of death. Debugging nightmare.
  • Examples: Linux, Traditional UNIX.
Real Talk
This is like having all your house systems in one room. Efficient? Yes. Safe? Not really.

2.2 Option B: Separated (Microkernel)

Architecture: Tiny kernel + Everything else in user space. The kernel ONLY does memory management, process scheduling, and Inter-Process Communication (IPC).

  • Why It's Beautiful: Driver crashes? Just restart it. System survives. Easier to debug and maintain. Modular and flexible.
  • Why It's Slower: Every operation = Message passing. Context switches everywhere. Performance overhead.
  • Examples: QNX (used in cars), MINIX, Mach.

2.3 The Hybrid Solution

The realization: Pure designs are beautiful but impractical. The Hybrid Approach puts critical services in kernel (speed when needed) and non-critical services in user space (safety when possible).

Example: Your Mac's XNU Kernel. Apple wanted Unix compatibility + Mach's elegance + Performance. Other Examples: Windows NT, Android (Linux + userspace framework).

The Lesson: Real-world OS design is about intelligent compromise.

3

Virtual Machines & Hypervisors

3.1 Virtual Machines

What if you want to run Windows AND Linux AND macOS... on ONE machine?

  • One physical computer
  • Multiple virtual computers
  • Each thinks it's alone on real hardware
  • All running simultaneously

Why this matters: Isolation (security sandboxes), Cloud computing foundation (AWS, Azure, Google Cloud), Testing without risk. One server = 50 virtual servers.

3.2 The Two Types of Hypervisors

The Hypervisor (Virtual Machine Manager) makes this possible.

  • Type 1: Bare-Metal Hypervisor: Runs DIRECTLY on hardware. Maximum performance. Used for Enterprise/cloud (VMware ESXi, Xen, KVM, Hyper-V, Apple Virtualization Framework).
  • Type 2: Hosted Hypervisor: Runs ON TOP of host OS. Easier setup, more overhead. Used for Development/testing (VMware Fusion, VirtualBox, Parallels).
  • Bonus: Containers (Docker) = Even lighter, share kernel.
4

Program vs. Process

4.1 PROGRAM = Recipe

  • Static file on disk
  • Passive, dormant, lifeless
  • Just bytes sitting there
  • Example: `/Applications/Safari.app`
  • One copy on disk

4.2 PROCESS = Cooking

  • Program IN EXECUTION
  • Active, dynamic, consuming resources
  • Has memory, CPU time, state
  • Example: Safari.app RUNNING
  • Can have multiple processes from one program (e.g. 3 Chrome windows = 3 processes with different PIDs).

The Transformation: Program (Disk) → OS Loads → Allocates Memory → Creates PCB → Process (RAM)

5

Process Control Block (PCB)

5.1 What is a PCB?

Your Mac runs 200+ processes. Each needs CPU time allocation, memory tracking, open files tracking, and current execution state. The solution is the Process Control Block (PCB).

PCB = A Data Structure for Each Process. Think of it as the OS's diary entry about each process.

It is stored in Kernel memory (protected, users can't touch it).

5.2 What does it store?

PCB Data
int pid; // Unique ID (like social security #)
int state; // New/Ready/Running/Waiting/Terminated
long program_counter; // Next instruction address
int registers[32]; // ALL CPU register values
void* page_table; // Memory mapping
int priority; // Scheduling importance
int cpu_time_used; // Total CPU time consumed
file* open_files[]; // All open file descriptors

The Process Lifecycle (5 States)

6.1 The 5 States

  • NEW → Being created, OS initializing PCB.
  • READY → Ready to execute, waiting in queue for CPU.
  • RUNNING → Currently executing on CPU (only 1 per core!).
  • WAITING (BLOCKED) → Waiting for I/O, event, resource.
  • TERMINATED → Finished executing, cleaning up.

6.2 State Transitions

  • 1. NEW → READY (Admission): Process creation complete, OS admits to ready queue, PCB fully initialized.
  • 2. READY → RUNNING (Dispatch): Scheduler selects this process, gets CPU time, starts executing.
  • 3. RUNNING → READY (Preemption): Time slice expires (timer interrupt) or higher priority process arrives. Goes back to ready queue.
  • 4. RUNNING → WAITING (Block): Process requests I/O (disk read, network request), waits for resource, or calls sleep().
  • 5. WAITING → READY (Wakeup): I/O operation completes, event occurs. Back to ready queue (NOT directly to running!).
  • 6. RUNNING → TERMINATED (Exit): Process completes execution, calls exit(), or crashes. OS reclaims resources.

6.3 Key Insights

Multiple processes can be READY (ready queue). Multiple processes can be WAITING (different events). Only ONE process per CPU core can be RUNNING.

7

Context Switching

7.1 What is Context Switching?

You have 200 processes, 8 CPU cores. Each process thinks it owns the CPU. The OS switches between them thousands of times per second.

Context Switch = Saving One Process, Loading Another

  • The Cost: Time: 1-10 microseconds (pure overhead!). Cache pollution. No useful work during switch.
  • When It Happens: Timer interrupt (every ~10ms), System call, I/O interrupt, Higher priority process ready.

7.2 The 2-Step Dance

Step 1: SAVE Current Process

  1. Save ALL register values to PCB
  2. Save program counter
  3. Save stack pointer
  4. Update process state (Running → Ready/Waiting)

Step 2: LOAD Next Process

  1. Select next process (scheduler decision)
  2. Load its register values from PCB
  3. Load its program counter
  4. Update state (Ready → Running)
  5. Jump to its next instruction
8

Real-world Implications

8.1 Real-World Impact

  • Performance Understanding: Why closing unused apps helps (fewer context switches). Why SSDs feel faster (less time in WAITING state). Why RAM matters (more processes fit in memory).
  • Security Understanding: Process isolation (one app crash doesn't kill system). VMs for security (Malware contained). Kernel/user mode (protection mechanisms).
  • Development Understanding: Why multi-threading matters. Why async programming helps (less blocking/WAITING). Why profiling shows states.

8.2 Mind-Blowing Facts

  • Right now, on your Mac: 200-300 processes running simultaneously.
  • 10,000+ context switches per second.
  • Every smooth interaction you have with your Mac is the result of this incredibly complex, precisely orchestrated dance happening thousands of times per second. You never see any of it. That's the magic of operating systems.
