1.3 TODAY'S JOURNEY

03

1.4 PREVENTION STRATEGY 1

04

Strategy: Make resources shareable when possible

Examples That Work:

✓ Read-only files - Multiple processes read simultaneously

✓ macOS Example: Multiple apps reading system config files

The Problem - Not Always Possible:

❌

Printers (can't print two jobs simultaneously)

❌

Disk blocks being written (data corruption!)

❌

Critical sections with shared variables

❌

Mutexes (defeats their purpose!)

Reality Check: Some resources are INHERENTLY non-shareable

Verdict: Can't rely on this alone!

1.5 PREVENTION STRATEGY 2

05

Two Approaches:

Approach A: Request All at Once

Approach B: Release Before Requesting

Advantages:

✓ Simple to implement

✓ Guarantees no hold-and-wait

Disadvantages:

❌

Low resource utilization (hoarding!)

❌

Starvation risk (big requests wait forever)

❌

Impractical - processes don't know all needs in advance!

  • Process must request ALL resources before starting
  • Get everything or get nothing
  • Example: Database transaction locks tables A, B, C all at once
  • Must release all held resources before requesting new ones
  • Start fresh with each request

Deadlock Avoidance Concepts

2.1 Hold and Wait Reality Check

06

Why might a text editor have difficulty using the "request

all resources at once" strategy?

A) Text editors are too simple

B) It doesn't know in advance which files the user will open

C) Text editors don't use resources

D) The strategy only works for database systems

ANSWER: B) It doesn't know in advance which

files the user will open

The text editor can't predict user behavior - which

files you'll open, when you'll open them, or how

many you'll need. This unpredictability makes

"request all at once" impractical for interactive

applications.

2.2 PREVENTION STRATEGY 3

07

Strategy: Allow OS to forcibly take resources from processes

Implementation Options:

Option A: If P1 needs resource held by waiting P2, preempt P2

Option B: If P1 can't get resource, preempt ALL its resources

When This Works:

✓ CPU time - Just context switch!

✓ Memory pages - Swap to disk

✓ macOS Virtual Memory - Preempts memory by swapping

When This FAILS:

❌

Printers - Can't "undo" half-printed page!

❌

Mutexes - State can't be saved/restored

❌

Database locks - Transaction consistency breaks

❌

File writes - Data corruption risk

The Challenge: Resource state must be savable and restorable

2.3 PREVENTION STRATEGY 4

08

Strategy: Impose total ordering on all resources

The Rule:

Why It Works - Proof:

For a cycle to exist: P1 ← R1 ← P2 ← R2 ← P3 ← R3 ← P1

But if P1 holds R1 and requests R3: R3 must have number > R1

And if P3 holds R3 and requests R1: R1 must have number > R3

CONTRADICTION! Both can't be true!

Therefore: No cycle possible!

Advantages:

✓ Simple and provably works

✓ Actually used in real systems!

Disadvantages:

❌

Must define global ordering (complex in large systems)

❌

Forces awkward programming (may need higher-numbered resource first)

  • Number all resources: R1, R2, R3, ..., Rn
  • Processes MUST request in increasing order
  • If you hold Ri, you can only request Rj where j > i

2.4 PREVENTION TRADE-OFFS SUMMARY

09

Key Insight: Prevention is CONSERVATIVE and PESSIMISTIC

The Question: Is there a SMARTER way? Next: Enter Deadlock Avoidance! 🎯

  • Assumes deadlock could happen
  • Restricts system even when deadlock wouldn't occur
  • Trades performance for guaranteed safety
  • Often leads to resource underutilization
Critical Issue

This scenario requires careful synchronization. Uncontrolled concurrent access will violate mutual exclusion, leading to race conditions, data corruption, or deadlocks.

2.5 WHEN PREVENTION MAKES SENSE

10

When to Use Prevention:

✓ Safety-Critical Systems

✓ Simple Systems

✓ Expensive Recovery

Example: Kernel locks often use resource ordering

→ Simple, fast, provably safe!

  • Medical devices (pacemakers, surgery robots)
  • Aerospace systems
  • Nuclear power control
  • Cannot afford ANY deadlock risk
  • Few resource types
  • Well-understood resource patterns
  • Static behavior
  • Where detecting/recovering is too costly
  • Real-time requirements
  • No rollback capability

3.1 THE PHILOSOPHY SHIFT - AVOIDANCE

11

Prevention vs Avoidance:

Prevention:

Avoidance:

The Key Insight:

Just because deadlock COULD happen doesn't mean it WILL!

Avoidance examines each request and asks:

"If I grant this, could deadlock occur later?"

If YES: Make process wait (even if resources available!)

If NO: Grant it immediately

  • Break the fundamental conditions
  • Rigid, restrictive
  • Like banning all cars from an intersection
  • "Allow danger, but navigate smartly"
  • All four conditions CAN exist
  • Make intelligent allocation decisions
  • Like having smart traffic lights!
• "Never let dangerous situations be possible"

3.5 INTRODUCING BANKER'S ALGORITHM

15

Bank Analogy:

remaining customers even if all demand at once!

Banker's Algorithm:

The Promise:

If we follow Banker's Algorithm, deadlock is IMPOSSIBLE!

Requirements:

1. Each process declares maximum resource needs

2. Processes request incrementally (not all at once)

3. System checks safety before every grant

  • Bank has limited cash (resources)
  • Customers have credit limits (max needs)
  • Customers take out loans (allocations)
  • Banker must ensure: Can always satisfy
  • The definitive deadlock avoidance algorithm
  • Works with multiple resource types
  • Makes smart lending (allocation) decisions
  • Ensures system never enters unsafe state

Banker's Algorithm & Ledger

4.1

The Banker's Ledger

For n processes, m resource types:

Available[m]:

Max[n][m]:

Allocation[n][m]:

Need[n][m]:

Key Relationship: Need = Max - Allocation
  • Vector showing currently available resources
  • Example: Available = [3, 2, 1] means 3 of R1, 2 of R2, 1 of R3
  • Matrix of maximum resource needs
  • Max[n][m] = max of resource type m that process n may need
  • Declared in advance!
  • Matrix of current allocations
  • Allocation[n][m] = resources of type m currently held by process n
  • Matrix of remaining needs
  • Need[n][m] = Max[n][m] - Allocation[n][m]
  • How much MORE each process might need

Resource Request Algorithm & Safety check

5.1 Detection Frequency Check

21

In Banker's Algorithm, the Need matrix represents:

A) Total resources each process will ever use

B) Resources currently allocated to each process

C) Remaining resources each process may still request

D) Resources available in the system

ANSWER: C) Remaining resources each process may still request

Need = Max - Allocation, showing how much MORE each process might

need to complete. It's not the max (that's Max matrix), not current (that's

Allocation), and not available (that's Available vector).

5.2 BANKER'S VS PREVENTION

22

Real-World Usage:

Why Banker's Isn't Used:

❌

Processes don't know max needs in advance

❌

Number of processes varies dynamically

❌

Overhead of safety checks

❌

Resources not fixed (devices plug/unplug)

  • Prevention: Widely used (resource ordering, especially)
  • Banker's: Rarely used in practice!

5.5 Key Takeaways

25

What We've Learned:

1. Prevention: Break ONE of Four Conditions

2. Resource Ordering is the Winner

3. Avoidance: The Smart Approach

4. Banker's is Beautiful but Impractical

  • Mutual Exclusion - Rarely feasible
  • Hold and Wait - Impractical for dynamic systems
  • No Preemption - Works for some resources
  • Circular Wait - Resource ordering WORKS! ✓
  • Simple, proven, actually used
  • Mathematical guarantee
  • Used in real kernels
  • Safe vs Unsafe states
  • Banker's Algorithm guarantees safety
  • But requires knowing the future...
  • Theoretical importance
  • Rarely used in real systems
  • Can't predict future needs

5.6 Practical Guidelines

26

Modern Best Practices:

1. Resource ordering for kernel locks

2. Lock-free data structures when possible

3. Design patterns that avoid hold-and-wait (GCD!)

4. Timeouts and watchdogs

5. User-initiated recovery (force quit)

5.7 Final Understanding Check

27

What key information must be known in advance for deadlock avoidance to work?

A) Current CPU utilization

B) Maximum future resource needs of each process

C) Priority levels of all processes

D) Execution time of each process

ANSWER: B) Maximum future resource needs of each process

Avoidance (Banker's Algorithm) requires the Max matrix - each process

must declare its maximum resource needs before execution. This lets the

algorithm predict worst-case scenarios and maintain safe states.
