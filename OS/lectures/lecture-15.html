1.3 Review of Producer-Consumer Problem

What is it?

Scenario

Producer: creates data → puts into buffer

Consumer: removes data → processes it

Buffer: limited size (bounded buffer)

Key Challenges

Producer waits if buffer is full

Consumer waits if buffer is empty

Mutual exclusion required for buffer access

Prevent race conditions

Synchronization Requirements

Only one process accesses buffer at a time

Producer sleeps when buffer is full

Consumer sleeps when buffer is empty

Waiting process wakes up when condition changes

  • A classic synchronization problem
  • Demonstrates process coordination using a shared buffer

1.4 Producer-Consumer Semaphore Solution

Consumer Code

item = buffer[out];

out = (out + 1) % n;

consumeItem(item);

Semaphores Used (n = buffer size):

mutex = 1 → Mutual exclusion,

empty = n → Empty slots in buffer and

full = 0 → Filled slots in buffer

Producer Code

item = produceItem();

buffer[in] = item;

in = (in + 1) % n;
while (true) {
wait(full);       // Wait if buffer is empty
wait(mutex);      // Enter critical section
signal(mutex);    // Exit critical section
signal(empty);    // One more empty slot
}
while (true) {
wait(empty);      // Wait if buffer is full
wait(mutex);      // Enter critical section
signal(mutex);    // Exit critical section
signal(full);     // One more full slot
}

1.5 The Dining Philosophers Problem

Problem Overview

Scenario

5 philosophers sit around a circular table

5 forks, one between each pair

Each philosopher alternates between: Thinking and Eating

To eat, a philosopher needs both left & right chopsticks

After eating, forks are released

Philosopher States

1.

Thinking: Not using forks

2.

Hungry: Tries to acquire forks

3.

Eating: Holds both forks

Key Issues Highlighted

Mutual exclusion of forks

Possible deadlock

Possible starvation

Fair resource allocation

  • A classic synchronization problem
  • Used to explain deadlock, starvation, and resource sharing
  • Originally proposed as a theoretical concurrency problem

Dining Philosophers Solutions

2.1 Dining Philosophers - Key Challenges

1. Mutual Exclusion

Only one philosopher can hold a chopstick

2. Deadlock

No one can proceed → System stuck

3. Starvation

4. Concurrent Access

Deadlock Scenario (Example)

T1: All philosophers pick up LEFT chopstick

T2: All try to pick up RIGHT chopstick

→ DEADLOCK (infinite waiting)

  • Chopsticks are shared resources
  • Simultaneous use is not allowed
  • All philosophers pick up left chopstick
  • Each waits for right chopstick
  • A philosopher may never get to eat
  • Neighbors keep eating alternately
  • Leads to unfair resource allocation
  • Multiple philosophers attempt to eat at once
  • Requires proper synchronization
  • Prevents conflicts and ensures progress
Critical Issue

This scenario requires careful synchronization. Uncontrolled concurrent access will violate mutual exclusion, leading to race conditions, data corruption, or deadlocks.

2.2 Naive Semaphore Solution (Incorrect)

Approach

Philosopher Code

eat();

Why This Fails

Deadlock is possible

Everyone waits for the right chopstick → DEADLOCK

  • One semaphore per chopstick
  • Each chopstick initialized to 1
  • All philosophers may hold one chopstick each
  • Each waits forever for the second one
● semaphore chopstick[5] = { 1, 1, 1, 1, 1};
void philosopher(int i) {

2.3 Solution 1 - Asymmetric Solution

Strategy

Why This Works

Philosopher Code (Asymmetric)

think();

// Last philosopher picks RIGHT first

// Others pick LEFT first

eat();

  • Use one semaphore per chopstick
  • Make one philosopher pick chopsticks in reverse order
  • Breaks symmetry in resource acquisition
  • Philosopher 4 picks chopsticks in opposite order
  • Circular wait condition is broken
  • Not all philosophers can hold one chopstick simultaneously
  • Deadlock is prevented
semaphore chopstick[5] = {1, 1, 1, 1, 1};
void philosopher(int i) {
while (true) {
if (i == 4) {
wait(chopstick[(i+1) % 5]);
wait(chopstick[i]);
} else {
wait(chopstick[i]);
wait(chopstick[(i+1) % 5]);
}
signal(chopstick[i]);
signal(chopstick[(i+1) % 5]);
}
}

2.4 Solution 2 - Allow Maximum N-1 Philosophers

semaphore room = 4; // Only 4 philosophers allowed

Philosopher Code

think();

eat();

Why This Works

Strategy: Allow at most 4 philosophers to try eating at the same time. Use an extra semaphore (room) to limit entry. Ensures

at least one philosopher can eat.

  • At most N−1 philosophers compete for chopsticks
  • At least one philosopher gets both chopsticks
  • Circular wait is impossible
  • Ensures progress (no deadlock)
semaphore chopstick[5] = {1, 1, 1, 1, 1};
void philosopher(int i) {
while (true) {
wait(room);           // Enter dining room
wait(chopstick[i]);   // Pick left chopstick
wait(chopstick[(i+1) % 5]);// Pick right chopstick
signal(chopstick[i]);
signal(chopstick[(i+1) % 5]);
signal(room);               // Leave dining room
}
}

2.5 Solution 3 - Using Mutex (All or Nothing)

semaphore mutex = 1; // Protects critical section

Philosopher Code

think();

eat();

Strategy: Use a global mutex to protect chopstick pickup. A philosopher must acquire both chopsticks atomically. Prevents

partial allocation of resources.

Pros: No deadlock. Easy to understand and implement. Ensures atomic

resource acquisition.

Cons: Poor concurrency. Only one philosopher can pick chopsticks at a

time. Reduces parallelism significantly.

semaphore chopstick[5] = {1, 1, 1, 1, 1};
void philosopher(int i) {
while (true) {
wait(mutex);              // Enter critical section
wait(chopstick[i]);       // Pick left chopstick
wait(chopstick[(i+1) % 5]); // Pick right chopstick
signal(mutex);            // Exit critical section
wait(mutex);              // Enter critical section
signal(chopstick[i]);     // Put left chopstick
signal(chopstick[(i+1) % 5]); // Put right chopstick
signal(mutex);            // Exit critical section
}
}

4.2 Introduction to Monitors

It is a high-level synchronization construct that provides:

Data Encapsulation (like a class/module)

Automatic Mutual Exclusion

Condition Synchronization (via condition variables)

Key Characteristics

1⃣ Encapsulation

2⃣ Mutual Exclusion

Only ONE process active inside the monitor at a time

3⃣ Structured Design

All synchronization logic in one place

  • Shared data is private to the monitor
  • Access only through monitor procedures
  • Enforced automatically by compiler/runtime
  • Clear separation of logic vs synchronization
  • Easier to read, debug, and maintain

4.5 Condition Variables

What are Condition Variables?

Syntax (Pseudo-code)

condition x, y; // Declaration

Operations

  • Special variables used for synchronization inside a monitor
  • Allow processes to wait for a condition to become true
  • Used with monitors, not standalone like semaphores
  • Process releases monitor lock
  • Goes to sleep on condition’s queue
  • Another process can enter the monitor
  • Wakes up ONE waiting process
  • If no process is waiting → signal is lost
x.wait();         // Wait on condition x
x.signal();       // Signal condition x
