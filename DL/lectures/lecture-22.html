Training Lifecycle

1.1 Pretraining (Unsupervised)

Large language models start with Pretraining. The model is trained on terabytes of raw text to predict the next token (causal language modeling). This step allows the model to learn syntax, facts, and reasoning representations from web-scale data in an unsupervised manner.

1.2 Supervised Fine-Tuning (SFT)

After pretraining, the model is a text-completer. To make it a useful assistant, we apply Supervised Fine-Tuning (SFT). We train the model on curated datasets of instruction-response pairs (e.g. "Explain gravity" -> "Gravity is..."), teaching it to act as an assistant.

1.3 RLHF & Alignment

To align the model's outputs with human preferences regarding helpfulness and safety, we use Reinforcement Learning from Human Feedback (RLHF). We train a reward model on human comparisons of outputs, and use algorithms like PPO (Proximal Policy Optimization) to adjust the model's weights.

Scaling Laws & Chinchilla Law

2.1 Power Laws of Compute, Data, and Parameters

Kaplan et al. showed that model performance follows power-law relationships with respect to compute budget, model parameter count, and training dataset size. Scaling these three variables in concert leads to predictable increases in model capacity.

2.2 Chinchilla Scaling Formula

Hoffmann et al. (the Chinchilla paper) identified that previous models were over-parameterized and under-trained. They found that for an optimal training budget, the parameter count and the number of training tokens should scale in a 1:1 ratio. For every doubling of compute budget, both model size and training tokens should double, yielding the optimal Chinchilla scaling law.

3

Training Infrastructure

3.1 GPU Memory Bottlenecks

Modern models with billions of parameters exceed the memory limits of a single GPU. Memory is consumed by model weights, optimizer states (like Adam's momentum/variance vectors), gradients, and intermediate activations, creating GPU memory bottlenecks.

3.2 Data Parallelism vs. Model Parallelism

To scale training across multiple GPUs, we split the workload:

  • Data Parallelism (DP): The model is replicated on every GPU, and each GPU processes a different slice of the batch. Gradients are averaged across GPUs at each step.
  • Model Parallelism (MP): The model's layers are split across multiple GPUs. This is necessary when model weights alone exceed a single GPU's memory.

4

Zero Redundancy Optimizer (ZeRO)

4.1 Optimizer Memory Redundancy

Standard Data Parallelism replicates optimizer states, gradients, and parameters across all GPUs, leading to significant memory redundancy. In fact, Adam optimizer states consume about 4 times more memory than the model weights themselves.

4.2 ZeRO Stage 1, Stage 2, and Stage 3 Partitioning

The Zero Redundancy Optimizer (ZeRO) eliminates memory redundancy by partitioning states across GPUs:

  • ZeRO-1: Partitions optimizer states across GPUs. Each GPU only updates a slice of the parameters.
  • ZeRO-2: Partitions both optimizer states and gradients across GPUs.
  • ZeRO-3: Partitions optimizer states, gradients, and the model parameters themselves. Parameters are gathered via communication calls only when needed for forward/backward steps, enabling the training of extremely large models.
